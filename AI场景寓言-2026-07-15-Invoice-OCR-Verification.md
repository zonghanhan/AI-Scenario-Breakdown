今天先讲 AI 工作流的第一块地基：机器怎样接住一张业务单据，又怎样在不确定时把决定权交还给人。后续无论做合同审查、简历筛选还是智能派单，都会反复用到这套思路。

# 2. 业务场景故事

## 《那张看起来完全正确的 8,600 元发票》

### 第一章：周一早上的 1,327 个附件

周一 9:10，连锁餐饮公司“禾味”的财务共享中心收到 1,327 张周末报销票据。财务主管林岚最先关心的并不是“能不能识字”，而是下午四点前能否把可报销单送进付款审批，同时别把重复票、抬头错误票和金额异常票放过去。

员工的提交方式很不整齐：有人上传原始 PDF，有人拍了一张桌面照片，还有人把三张出租车票拼在一张长图里。系统先做的也不是识别，而是“收件检查”：

- 文件是否为允许的 PDF、OFD、JPG、PNG，是否损坏或加密；
- 图片有没有严重模糊、反光、缺角，方向是否颠倒；
- 一份文件里有几张票，每张票的边界在哪里；
- 报销单号、提交人、费用类型等上下文是否齐全。

不合格文件不会硬着头皮往下跑。加密 PDF 直接提示员工重新上传；严重缺角的图片退回补拍；一张图中有多张票，则先切成独立票据并保留“来自同一张原图”的关系。这个入口看似不智能，却决定了后面一半的准确性——垃圾输入不会因为接了模型就变成可靠数据。

### 第二章：机器读出的不是一句话，而是一组待验证的字段

通过收件检查后，系统开始处理一张设备维修发票。图像预处理先自动旋转、裁边、去阴影，再交给 OCR（光学字符识别，也就是“从图片里读出文字和位置”）。

OCR 返回的不只是整段文字，而是一组结构化字段：

- 发票号码：`2460……3187`
- 开票日期：`2026-07-12`
- 购买方名称：`禾味餐饮管理有限公司`
- 销售方名称：`启明制冷设备服务部`
- 不含税金额：`8,113.21`
- 税额：`486.79`
- 价税合计：`8,600.00`

每个字段还带着它在原图中的坐标和一个置信度（模型对自己这次识别有多大把握的评分）。“价税合计”很清楚，置信度高；销售方税号有一位被印章盖住，`8` 和 `B` 很难分，置信度低。

产品经理小周原来想过一个简单方案：“整张票平均置信度超过 95% 就自动通过。”林岚当场否决了。发票备注识错一个字可能无关紧要，税号、号码、金额错一位却可能让整张票不可用。于是系统采用字段分级：金额、票号、税号、日期是关键字段，门槛更高；备注、地址等普通字段可以容忍较低置信度。机器此时只有“抄录权”，还没有“放款决定权”。

### 第三章：8,600 元对上了，为什么还是被拦住

系统接着把字段送进规则层。第一组是票面内部校验：不含税金额加税额是否与价税合计一致，日期是否真实存在，税号格式是否合法，购买方名称是否与企业主数据一致。

第二组是跨系统核验：

- 用票号、日期、金额等要素查询合规的验真渠道，确认票据状态；
- 在报销历史库中查找相同票号，防止同一张票重复报销；
- 对照采购单、维修工单和供应商档案，检查“谁买的、买了什么、为什么买、该付多少”；
- 根据公司政策检查费用类型、报销期限和审批额度。

这张 8,600 元的发票算术完全正确，票面也很清晰，却命中了“疑似重复”：两天前另一个门店提交过相同票号和金额。系统没有直接判定员工舞弊，因为也可能是供应商重复发送附件、员工选错报销主体，或者历史数据重复导入。它只把票据状态改成“高风险待复核”，并附上命中理由和关联单据。

走到这里，小周才真正意识到，这不是一个“把图片变成文字”的 OCR 小功能，而是一套发票自动识别与核验系统。识别只回答“票上写了什么”，核验才回答“这些内容是否可信、是否符合当前业务”。

### 第四章：不是所有异常都该排进同一条人工队伍

午前，系统已经把票据分进四条路径：

1. **自动通过**：关键字段高置信、验真成功、业务规则全部通过，进入原有费用审批流；
2. **快速复核**：只有一两个低置信字段，复核员对照原图修改或确认；
3. **风险复核**：重复票、红冲状态、订单金额不一致等，需要查看关联证据，必要时找业务人员说明；
4. **退回补件**：原图不可读、发票缺页或必要凭证缺失，退给提交人处理。

复核工作台没有只展示一个“模型说错了”的红点。左边是原图，并高亮可疑字段；右边是识别值、置信度、规则命中原因、验真结果和历史相似单据。复核员可以“确认原值”“修改字段”“判定不可报销”或“升级给主管”，每一步都记录操作者、时间、修改前后值和理由。

这里有个很现实的效率问题：如果把所有低于统一阈值的票都交给人，模型上线后财务可能比以前更忙。因此团队按“错误后果”分流。50 元出租车票的非关键字段模糊，可以低风险放行；8,600 元维修票命中重复，即使 OCR 每个字都认对，也必须人工判断。人工资源应该花在风险上，而不是平均花在每张票上。

### 第五章：下午两点，外部验真接口超时了

14:05，外部验真服务连续超时。一个脆弱的设计会让全部报销卡死；另一个危险的设计则会在接口失败时默认全部通过。

团队预先设计了降级策略：

- 已完成 OCR 和内部规则校验的票据继续处理，但标记为“待验真”，不进入最终付款；
- 请求进入带幂等键的重试队列——同一张票反复重试也只生成一条核验任务；
- 超过重试上限后进入异常队列，由值班人员处理，不无限占用主流程；
- 临近付款批次的高优先级票据可转人工登录官方渠道核验，结果仍需回填并留痕；
- 恢复后系统自动补偿执行，对失败期间的票据批量补验。

同一时间，还有一张图片因反光导致模型完全找不到金额区域。系统没有让模型“猜一个最像的金额”，而是明确返回“不可判定”，要求补拍。能说“不知道”，是财务 AI 比“每次都给答案”更重要的能力。

### 第六章：上线复盘，95% 准确率为什么还不够用

第一周复盘时，算法同事汇报“字符识别准确率 95%”。林岚追问了三个更接近业务的问题：

1. 关键字段全部正确的票据比例是多少？
2. 自动通过的票里，有多少后来被发现不该通过？
3. 人工每复核一张票要多久，最常改的是哪些字段？

团队最终把指标拆成四层：

- **识别质量**：关键字段准确率、字段缺失率、按票种和图像质量分层的表现；
- **核验质量**：风险召回、错误拦截率、自动通过后的漏错率；
- **流程效率**：自动通过率、人工复核率、平均处理时长、异常积压量；
- **业务结果**：单票综合处理成本、重复报销拦截金额、付款退回率、用户补件率。

阈值也不再追求“一刀切”。阈值调高，漏错可能减少，但人工队列和等待时间会上升；阈值调低，自动化率变漂亮，纠错和财务风险却可能变贵。团队每周抽检一部分“自动通过”的票，用复核修改记录建立错误样本集，再按票种、供应商和拍摄质量调整规则与模型。

到下午四点，那张 8,600 元发票被人工确认是供应商重复发送、员工重复提交，报销申请被撤回。系统没有替财务做最终定性，但它从 1,327 张票中准确地把最值得人看的那一张推到了前面——这才是这套系统真正创造的价值。

# 3. 系统正解

这套系统的核心数据流是：文件接入与质量检测 → 图像预处理和票种分类 → OCR 字段抽取 → 票面规则、业务规则与外部验真 → 风险分流 → 人工复核 → ERP/审批系统回写。故事第一、二章对应输入治理和识别模块，8,600 元票据被拦截对应规则引擎与跨系统核验，第四章对应工作流编排和复核台。外部接口超时后的重试、待验真状态和补偿任务属于容错与降级模块，而复核修改记录会进入评估数据集，支持后续迭代。系统的目标不是让 OCR 单点分数最高，而是在可审计的前提下，用合理成本把正确数据送进付款流程，并把高风险、不确定样本交给合适的人。

- **写需求文档时**：不要只写“支持发票识别”，要列出票种、必填字段、每个字段的输出格式、异常状态和下游回写条件。
- **做方案评审时**：问清楚置信度是整票级还是字段级，外部验真超时、重复请求和人工修改后如何处理。
- **设计指标时**：同时看关键字段准确率、自动通过漏错率、人工复核率和单票综合成本，不能只看模型准确率。
- **跨团队沟通时**：与算法确认低置信和不可判定的输出，与财务确认风险权限，与工程确认状态机、幂等、审计日志和补偿机制。

# 4. 类比锚定

可以把这套系统类比成“机场入境行李分拣与安检”。行李先被规范接收、扫描和识别，再按规则判断去普通转盘、开箱复检还是异常处置；机器负责高吞吐筛查，人负责高风险判断。这个类比的关键不是“都有传送带”，而是都在有限人工下，根据风险把大量对象分流。

| 机场行李分拣与安检（旧经验） | 发票识别与核验系统（新系统） | 对应关系为什么成立 |
|---|---|---|
| 行李标签与航班号 | 报销单号、文件 ID、提交人上下文 | 都是对象进入流程后的唯一身份与路由依据 |
| 行李尺寸、破损检查 | 文件格式、清晰度、缺角和方向检测 | 都在正式处理前拦截不合格输入 |
| X 光扫描成像 | OCR 与版面分析 | 都把肉眼可见对象转换为可供系统判断的信号 |
| 禁运品与转机规则 | 金额勾稽、重复票、抬头和费用政策规则 | 都是确定性规则，命中后触发特定路径 |
| 普通行李自动送往转盘 | 高置信、低风险票据自动进入审批 | 都让大多数标准对象低成本通过 |
| 可疑行李开箱复检 | 低置信或高风险票据人工复核 | 都把有限人工集中到机器无法安全判定的对象 |
| 行李转错后的追踪记录 | 原图、识别值、修改值和操作日志 | 都依赖全链路留痕定位责任和恢复处理 |
| 扫描设备故障时转人工通道 | OCR/验真服务故障后的排队、补偿与人工核验 | 都需要降级，不能让主流程无声放行或永久堵塞 |

# 5. 类比边界与系统要点

## A. 系统核心要点总结

- **数据输入要求与格式规范**：先明确支持的票种、文件类型、页数、大小、清晰度和必需业务上下文；原始文件、切票结果与结构化字段要能关联追溯。
- **模型能力边界与置信度**：OCR 擅长从合格图像提取文字，不等于理解票据真假或费用合理性；置信度应落到字段级，并按金额、税号等关键程度设置不同策略。
- **人工介入条件与流程**：低置信关键字段、验真失败、重复票、业务单据不一致和高金额风险应触发不同复核队列；人要看到原图、证据、命中规则和可执行动作。
- **容错与降级机制**：模型不可用时可排队或转人工，验真不可用时应进入“待验真”而非默认通过；重试要幂等，失败要有异常队列和恢复后的补偿任务。
- **成本结构**：成本不只有 OCR/API 调用费，还包括图片存储、规则维护、系统集成、人工复核、错误付款后的追回与审计；应按“每张成功入账票据的综合成本”评估。
- **持续评估闭环**：保留人工修改和抽检结果，按票种、供应商、图像质量监控效果；阈值变化要经过离线回放和小流量验证，避免只提高自动化率却放大风险。

## B. 类比在哪里会失效

1. **机场规则大多相对稳定，AI 输出却带概率性。** 同一版式在反光、印章遮挡或模型版本变化后可能得到不同结果，不能把一次“扫描通过”理解为确定事实。
2. **行李风险常可由实物复检确认，发票真实性依赖外部数据和业务语境。** OCR 看不出一笔维修是否真实发生，必须结合验真渠道、工单、采购单、供应商和历史报销。
3. **一条安检规则通常对所有行李一致，发票阈值要按风险分层。** 票种、金额、费用类别和客户政策不同，自动通过条件也不同；不存在一个长期通用的 95% 阈值。
4. **传送带扩容主要是设备问题，AI 系统扩容还会改变人工与数据分布。** 自动化后留下的都是更难样本，人工单票耗时可能反而上升，不能用试点阶段平均耗时直接推算规模化 ROI。
5. **安检类比无法覆盖持续迭代和模型漂移。** 新版数电发票、供应商模板变化、拍摄设备变化都会让历史表现失真，产品必须持续抽检、回放、更新规则并治理模型版本。

# 6. 深度业务洞察

1. **业务侧：用户嘴上要“自动识别”，真正要的是“可入账且少退回”。** 如果只交付一个 JSON，财务仍要手工核对、复制到 ERP，整体处理时间不会明显下降。立项时应画完整流程，找到最耗时的异常，而不是把 OCR 接口调用成功当作项目完成。

2. **技术侧：整票平均准确率最容易制造虚假的安全感。** 一张票几十个字段，地址识错与金额识错后果完全不同；而“每个字段都挺准”也不代表“关键字段同时全对”。验收集必须覆盖模糊、旋转、盖章、截图、拼图、不同票种和新模板，并按关键字段与整票可用率报告。

3. **运营侧：自动化率提高后，复核队列会越来越“难”。** 简单票被机器拿走，人工剩下的多是证据缺失、跨单据冲突和高风险案例，单票处理时间不会线性下降。排班与 ROI 模型要使用分流后的真实复核时长，并给复核员设计快捷键、证据并排和批量处理能力。

4. **合规侧：可解释不是展示一句模型理由，而是保留证据链。** 至少要关联原始文件、模型与规则版本、字段值及坐标、外部核验结果、人工修改和最终去向；敏感票据还要控制访问权限、传输与存储范围、保留期限。使用第三方云 OCR 前，应确认数据是否跨境、是否用于厂商训练以及删除机制。

5. **商业侧：最贵的往往不是调用模型，而是“最后一公里”。** 客户主数据不干净、ERP 字段各不相同、财务政策频繁变化，都会带来实施和长期维护成本。定价与 ROI 评估应按票量、票种复杂度、系统集成数和人工节省拆开计算，同时计入误拦导致的付款延迟与漏拦导致的追回成本。

   适合负责“验真”需求边界的 PM。它能帮助你区分“票面识别”与“权威查验”；产品接入前仍需让法务、财务和工程确认适用渠道、调用方式、频率限制与授权条件，不能把网页查验能力直接等同于开放 API。

## 30 分钟动手小练习

6. **画一张“报销票据状态机”**  
   找一张已脱敏的发票样例，用表格写出 8 个关键字段及格式；再画出“待上传 → 识别中 → 待验真 → 自动通过 / 待复核 / 退回补件 → 已回写”的状态流。为每次状态转换补上触发条件、失败去向和责任人。完成后自问：验真超时会去哪、人工改金额后是否重新跑规则、重复请求会不会生成两笔记录？这三个问题能快速检验你的方案是不是一个可运行的工作流。
