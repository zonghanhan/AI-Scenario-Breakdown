# 1. 元信息

- **日期**：2026-07-17
- **场景名称**：工程 AI 派单 / Field Service Dispatch
- **故事名称**：暴雨夜，那条派给河对岸的电梯工单
- **故事类型**：博弈型
- **难度**：入门
- **系列位置**：AI 工作流系列 4/5
- **前置概念**：
  - **地理围栏（Geofencing）**：在地图上画圈，判断师傅「在哪儿」「离工地多远」，是派单的距离底座
  - **技能标签（Skill Tag）**：电梯、消防、弱电、高空作业等资质，决定「谁能接这活」
  - **VRP（车辆路径问题）**：在一堆工单和一堆师傅之间，算「谁先去哪儿」更省总路程、更守 SLA——本质是运筹优化，不是简单比直线距离

上一篇智能工单分配解决的是「客服台分给谁」，这一篇工单已经出了大楼：**师傅在雨里开车，派错一个人，代价是困梯、漏水、罚款，还有明天的投诉**。工程派单是多方博弈：客户要快到、师傅要合理单量、公司要控成本、算法要背锅。

# 2. 业务场景故事

## 《暴雨夜，那条派给河对岸的电梯工单》

### 第一章：17:42，23 张红牌工单压在调度大屏上

台风外围雨带压过江城。设施维保公司「腾达维保」调度中心，运营总监周原盯着大屏，声音发紧：「困人电梯两张，商场漏水五处，剩下全是 P1。今晚 SLA——困人 45 分钟到场，漏水 90 分钟。」

产品经理韩溯刚入职两周，被周原拽到工位：「你上周上线的『智能就近派单』，先把这 23 张红牌清掉。明早我要给甲方交响应率报表。」

大屏弹出第一条自动派单结果：工单 **#E-4407**，某写字楼 18 层客梯困人 4 名，被派给 **李岩**——河对岸浦东网格的电梯学徒，直线距离 11.2 公里，驾车导航 38 分钟。

周原拍桌：「赵师傅就在对岸 800 米，为什么没派他？」

算法工程师许蔷调出日志：`assign_to=LiYan, reason=nearest_available`。赵师傅状态显示 **忙碌**，正在 600 米外一家便利店做「照明灯更换」——一张标价 80 元的 P3 小单。

韩溯喉咙发紧。困人梯等学徒过河，甲方合同里写的是 **到场超时每分钟罚 200 元**。

「就近派单，」周原冷笑，「近的是地图上的点，不是救人的点。」

---

### 第二章：第一局博弈——「最近」还是「最会」？

当晚复盘，三方挤在会议室。许蔷在白板上写三个目标：

「系统默认优化 **总行驶里程最短**，所以选李岩。赵师傅不是不近，是状态被小单占住了。」

韩溯问：「困人、漏水算 P0/P1，能不能跟换灯泡分开？」

许蔷点头：「可以上 **技能路由**：电梯困人必须 `elevator_certified`；消防喷淋必须 `fire_safety`；普通巡检谁都能接。」

**第一个产品二选一**摆在桌上：

- **方案 A**：P0/P1 一律 **技能优先**，在合格师傅里选 ETA（预计到达时间）最短；
- **方案 B**：全等级继续 **距离优先**，技能只做过滤，不合格再换下一近的人。

周原要 A：「困人不能赌学徒第一次独立处理。」韩溯担心：「持证师傅就 14 人，今晚全城堵车，会不会更慢？」

最终拍板：**P0 技能优先 + 可打断低优先级在手单；P2/P3 距离优先**。许蔷连夜改权重。

周六 0:15，#E-4407 重算，赵师傅被中断小单，12 分钟到场，困人 28 分钟解除。周原松口气：「这才像派单。」

韩溯心想：问题解决了。

他没想到，下一波工单会把「技能优先」变成「把高手钉死在堵车路上」。

---

### 第三章：以为稳了——三张 P0 把唯一的高级电梯组堵在长江大桥

凌晨 1:40，雨更猛。系统连续涌入三张 P0：

1. 商场客梯困人（2 人）；
2. 医院货梯故障，手术室物料下不来；
3. 地铁换乘站扶梯急停，晚班客流积压。

持证电梯师傅今晚只有 **赵师傅、孙姐、老吴** 三人值班。技能优先算法做了「正确」分配：三人各接一单，全部指向不同城区。

问题是：**三单同时过长江大桥——导航全红，ETA 从 22 分钟飙到 71 分钟**。

更离谱的是网格 **河西** 有一张 P2「空调漏水」，离待命中的 **周师傅** 只有 1.1 公里。周师傅有弱电证，没有电梯证，系统死规矩不放行。漏水蔓延到配电间，甲方手动升级成 P0，可电梯组没人能抽身。

周原电话炸响：「你们派了三个高手，结果三个都在桥上排队？」

许蔷拉日志，声音发涩：「算法假设 **同时出发、互不影响**。现实是 **路网容量** 和 **工单时间窗** 会互相卡死。」

**第二个产品二选一**更尖锐：

- **方案 C**：P0 继续单工单最优，不管全局；
- **方案 D**：引入 **批量调度（Batch Dispatch）**，每 15 分钟把同一片区的 P0/P1 一起重算，允许「近处非持证师傅先到场围挡 + 持证师傅接力」。

韩溯和周原吵了四十分钟。周原要 C：「规则简单，出了事好解释。」韩溯坚持 D：「暴雨夜不是平时，要允许 **人机协同的临时编组**。」

妥协方案落地：**P0 允许「双阶段派单」**——先派最近任意师傅「到场稳控」（断电围挡、安抚被困），再派持证师傅「处置解锁」；系统把两张子工单绑成一组，KPI 按整组到场算。

老吴先 8 分钟赶到地铁扶梯做围挡，赵师傅 34 分钟到场复位；医院货梯由孙姐接力，周师傅被临时授权「弱电+围挡」先控水。三张红牌在 90 分钟内全部降级。

周原抹了把脸：「这才叫博弈——不是谁更近，是 **谁先稳住局面**。」

---

### 第四章：调度台变成牌桌——五方筹码如何摆？

周日后半夜，韩溯把派单逻辑画成五枚筹码：

| 筹码 | 代表什么 | 调太高会怎样 |
|---|---|---|
| SLA 紧迫度 | 困人 > 漏水 > 巡检 | 小单永远没人接，师傅怨单量不均 |
| 技能匹配 | 持证才处置 | 高手堵在半路，近处的人干看着 |
| 行驶 ETA | 导航实时路况 | 只看直线距离会翻第一章的车 |
| 备件可达 | 工单要带的关键配件 | 人到了缺零件，等于白跑 |
| 公平性 | 夜班单量、工时平衡 | 只追 SLA，同一师傅被榨干 |

新调度台不再只显示「派给了谁」，而是给调度员 **三行方案对比**：

- **方案 1（SLA 优先）**：预计全局超时最少，但孙姐连轴 6 小时；
- **方案 2（成本优先）**：总里程最短，医院货梯可能晚 18 分钟；
- **方案 3（均衡）**：允许一张 P2 顺延 30 分钟，换取扶梯先复位。

值班调度员陈姐可以一键选方案，也可以手动拖单——每次覆盖都记原因，回流训练。

许蔷强调：「AI 在这里不是独裁者，是 **发牌员**。它算出三组牌面，人拍板。」

韩溯把 **异常降级** 写进 PRD：

- 导航 API 超时 → 用上次路况缓存，置信度降一级，P0 必须人工确认；
- 师傅 GPS 漂移 → 连续 3 次定位跳变标 `location_untrusted`，不自动派新单；
- 备件仓 WMS 离线 → 禁派「强依赖备件」工单，或改派「备件车」在库的师傅。

---

### 第五章：GPS 集体失联——算法沉默时，网格班表说话

周一凌晨 3:10，暴雨致基站拥塞，**37% 师傅 GPS 延迟超过 8 分钟**。系统仍按「旧位置」派单，差点把老吴派回已关闸的高架。

许蔷触发 **红色降级**：自动派单暂停，切换 **网格预置班表**——每个片区谁值班、谁背 P0，事先写在纸上；调度台改电话确认：「你现在在哪？能接困人吗？」

陈姐带四名调度员分片区外呼，6 分钟恢复可见运力。自动派单在 GPS 恢复率回到 90% 后逐步重启，重启后 **批量重算** 降级期间积压的 41 张单。

晨会复盘，周原问韩溯：「这套东西到底省了什么？」

韩溯摊开账：

- 自动派单覆盖率 **54%**——故意没做 100%，暴雨夜证明 **全自动是幻觉**；
- P0 平均到场从 41 分钟降到 29 分钟，但 **调度员人均决策** 从每晚 12 次升到 19 次——人在关键局介入更多；
- 总行驶里程降 9%，油费省了，但 **双阶段派单** 带来子工单人力，要算进综合成本；
- 最大收益是 **误派投诉**：从每月 7 起降到 2 起，甲方罚款少了一截。

许蔷补刀：「下周路网施工，ETA 模型又要重训。派单是 **运营博弈系统**，不是上线即完工。」

韩溯写下结论：**工程 AI 派单 = 实时位置 + 技能约束 + 路况 ETA + 备件约束 + 批量优化 + 人工选方案 + 降级班表**。地图上的一个点，从来不是故事的全部。

# 3. 系统正解

工程 AI 派单的数据流是：**工单接入与分级 → 客户/设备主数据与 SLA 策略 → 师傅状态与技能/位置/备件拉取 → 单工单或批量优化派单 → 导航 ETA 回写 → 到场/完工/备件消耗回传 → 调度覆盖日志回流**。

- **第一章**对应位置服务与「忙碌状态」——小单占住高手，会让「最近可用」变成笑话。
- **第二章**对应技能过滤与优先级打断——P0 有权插队，但要写清中断低单的成本谁担。
- **第三章**对应批量调度与双阶段派单——单工单最优在拥堵路网下会全局翻车。
- **第四章**对应多目标优化与调度台——AI 出方案，人拍板，覆盖留痕。
- **第五章**对应位置降级与网格班表——定位不可信时，宁可慢也不能盲派。

# 4. 类比锚定

**类比：外卖平台的「骑手调度」，但每单要求「必须带某把钥匙的骑手」才能送达。**

外卖也面临「近不近」和「快不快」的拉扯；工程派单多了 **技能硬约束** 和 **备件耦合**，更像「只有能开某种锁的骑手，才能送这份餐」。

| 外卖骑手调度（旧经验） | 工程 AI 派单（新系统） | 映射关系 |
|---|---|---|
| 商家出餐时间 | 工单分级、甲方 SLA 截止 | 决定这单有多「饿」 |
| 骑手实时 GPS | 师傅位置与在线状态 | 派单的距离底座 |
| 骑手评分/车型 | 技能证、工龄、安全等级 | 不是谁都能接 |
| 骑手餐箱里的备餐 | 车上备件、仓库库存 | 人到场没零件等于白跑 |
| 高峰期拼单顺路 | 批量 VRP、同片区合并 | 避免三人同时堵一座桥 |
| 骑手拒单/转单 | 调度员手动改派 + 原因码 | 人的否决权与审计 |
| 导航 API 挂了 | GPS 降级 + 网格班表 + 电话确认 | 定位不可信要保守 |

# 5. 类比边界

## A. 系统核心要点

- **输入不止距离**：工单类型、设备型号、所需证件、备件 BOM、客户是否困人、路况、天气都会改派单结果。
- **模型边界**：ETA 预测是概率估计，暴雨、封路、事故会变；技能匹配是硬约束，不能靠模型「猜师傅会不会」。
- **人工介入条件**：P0 批量冲突、GPS 不可信、备件不确定、客户投诉升级、双阶段第二棒——都应走调度台或人工确认，不能硬自动。
- **降级策略**：定位/导航/WMS 任一关键依赖失败时，暂停盲派，切预置班表；恢复后批量重算积压单。
- **成本结构**：不算误派导致的罚款、二次上门、客户流失，只谈「少跑几公里」，ROI 会骗人。

## B. 类比失效点

1. **外卖订单相对同质，工程工单设备型号千差万别。** 同一「电梯故障」，品牌不同所需技能与备件完全不同，不能共用同一套标签。
2. **外卖可转单给任意空闲骑手，工程有强合规。** 高空、消防、压力容器未持证到场，出事故是法律责任，不是差评能覆盖的。
3. **外卖高峰可加价刺激运力，维保合同价固定。** 你不能临时加价让师傅过桥，只能靠排班、补贴和优先级博弈。
4. **外卖顾客能实时看骑手在哪，甲方只看 SLA 报表。** 产品要设计「可对外证明」的到场证据链：GPS 轨迹、打卡、照片、通话记录。
5. **类比掩盖「师傅是人不是点」。** 连续夜班、情绪、家庭距离都会影响接单意愿；纯算法最优可能遭集体拒单，要有公平性筹码。

# 6. 复盘室

### 复盘 1

- **周原说**：「就近派单最公平，师傅没怨言，先把距离算法调到极致。」
- **真相是**：工程场景里 **「忙」不等于「不能打断」**；低优先级小单占住高手，会让 P0 的「近」变成全局的「远」。
- **如果你做 PM**：状态机要区分 **可中断 / 不可中断** 在手单，并写清中断规则与补偿——否则运营会用「手动改状态」掏空系统。

### 复盘 2

- **许蔷说**：「每张 P0 都做技能优先单工单最优，总指标最好。」
- **真相是**：单工单最优在 **路网容量有限** 时会互卡，就像三人同时上桥；需要批量视角或双阶段接力。
- **如果你做 PM**：暴雨/大促要单独做 **压力场景回放**；实验室里的平均 ETA，经不起一座桥的红灯。

### 复盘 3

- **韩溯说**：「GPS 延迟几分钟没关系，先自动派，师傅会自己赶。」
- **真相是**：定位延迟 8 分钟在暴雨夜等于 **盲人开车**；误派比晚派更贵，降级要默认停自动、启班表。
- **如果你做 PM**：给位置服务画 **可信度分数**，低于阈值禁止自动 P0；恢复后必须有 **积压单重算** 任务，不能假装降级期间没发生。

# 7. 学习资料

1. **[Salesforce Field Service：Skills-Based Scheduling](https://help.salesforce.com/s/articleView?id=service.fs_skills_based_scheduling.htm)**  
   **适合**：第一次写维保派单 PRD 的 PM。  
   **解决什么**：技能、区域、时间窗在成熟现场服务系统里如何配置，避免从零造名词。

2. **[Google OR-Tools：Vehicle Routing Problem 入门](https://developers.google.com/optimization/routing/vrp)**  
   **适合**：需要和算法对齐「批量派单」边界的 PM。  
   **解决什么**：理解「单工单最近」与「多工单总路径最优」不是一回事，便于评审方案 C 还是 D。

3. **[ServiceNow Field Service Management 产品概览](https://www.servicenow.com/products/field-service-management.html)**  
   **适合**：做 ToB 维保、要对接甲方工单系统的 PM。  
   **解决什么**：现场工单状态机、移动端回传、SLA 计时在行业产品里长什么样。

4. **[Uber Engineering：Marketplace Dispatch 系列博客](https://www.uber.com/blog/engineering/)**  
   **适合**：想理解「派单是供需博弈」的 PM（在博客内搜 dispatch / marketplace）。  
   **解决什么**：行业案例说明 ETA、供需平衡、人工干预在规模化调度里的位置——别把它想成「算个距离」就完事。
