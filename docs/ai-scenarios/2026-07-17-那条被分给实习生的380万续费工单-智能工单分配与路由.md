# 1. 元信息

- **日期**：2026-07-17
- **场景名称**：智能工单分配与路由 / Intelligent Ticket Routing
- **故事名称**：那条被分给实习生的 380 万续费工单
- **故事类型**：侦探型
- **难度**：入门
- **系列位置**：AI 工作流系列 3/5
- **前置概念**：
  - **意图分类（Intent Classification）**：判断用户「想干什么」——报障、询价、投诉还是续费谈判
  - **技能路由（Skill-based Routing）**：按坐席专长把工单分给「最合适的人」，而不是「最闲的人」
  - **SLA（服务等级协议）**：承诺多久必须首次响应、多久必须解决，超时会有罚款或丢单

上一篇商品翻译解决的是「内容对不对」，这一篇进入客服运营：**同一条工单，分给不同的人，结果可以差一个数量级**。智能分配不是「找个 AI 自动派活」，而是在速度、质量、成本和背锅之间做路由。

# 2. 业务场景故事

## 《那条被分给实习生的 380 万续费工单》

### 第一章：周一 9:17，华创的工单进了实习生队列

企业软件公司「云栈科技」的客服主管唐玫盯着大屏，脸色发白。华创集团——年合同 380 万、下周续费谈判的关键客户——凌晨提交了一条标题为「系统又崩了，续费还谈不谈」的工单。

工单却被路由给了入职两周的实习生小柯。客户采购总监在工单里写：**「两小时内没人回，我们走竞品。」**

产品经理方栎被唐玫从会议室拽出来：「你的智能分配上周刚全量上线，今天第一条大客户工单就翻车。续费窗口本周五关账，这条 SLA 是 30 分钟首响。」

方栎打开后台，路由日志只有一行冷冰冰的结论：`route_to=junior_queue, reason=load_balance`。

「负载均衡把 VIP 扔给实习生了？」唐玫咬牙，「客户要的是能拍板的人，不是会查知识库的人。」

方栎心里一沉——这不是某次偶发失误，而是路由逻辑根本没把「谁」和「什么事」放在一起看。

---

### 第二章：先抓意图，再抓人——第一版方案看起来很合理

当天下午复盘，算法工程师陆沉在白板上画了三步：

「工单进来，先做**意图分类**——模型读标题和正文，判断是技术故障、账单问题、功能咨询还是商务续费。再查**客户主数据**，看是不是大客户。最后做**技能匹配**，把技术单给工程师坐席，把续费单给客户成功经理。」

方栎点头：「比纯轮询强。我们能不能今晚就改？」

陆沉提醒：「意图模型在测试集上准确率 91%，但测试集里『续费+报障』混合工单很少。」

**第一个产品二选一**摆在桌上：

- **方案 A**：意图置信度 ≥ 85% 才自动路由，否则进人工分诊台；
- **方案 B**：不管置信度，先按「最空闲坐席」派单，追求平均等待时间最短。

方栎选了 A。「大客户丢不起，宁可多一道人工，不能乱派。」

当晚 hotfix 上线。路由规则变成：识别为 `renewal` 且客户等级为 VIP → 客户成功队列；识别为 `incident` → L2 技术队列；置信度低于 85% → 分诊台。

唐玫松了口气：「至少不会再把华创扔给实习生了。」

她没想到，真正的坑在「意图认错了，后面全对也没用」。

---

### 第三章：以为修好了——愤怒续费单被当成普通账单咨询

周三下午，华创又出一条工单。标题是「发票开错了，你们还想不想续」。正文前半段骂了三百字发票税率，后半段写「下周合同评审，今天给说法」。

新路由把它标成 `billing`，置信度 92%，自动分给了账单专员苏晴。

苏晴十分钟后回复：「已为您更正发票抬头，请查收邮件。」

华创采购总监直接电话炸到销售 VP：**「我们在谈几百万续费，你们派个开票的来敷衍？」**

方栎和陆沉连夜拉日志，像侦探查案：

1. **标题触发词**：「发票」权重过高，压过了「续费」；
2. **情绪信号**：大量负面词被模型当成「抱怨账单」，没抬升到商务风险；
3. **客户标签**：华创虽是 VIP，但 `billing` 路径不查客户等级，只查技能表；
4. **坐席状态**：苏晴当时最闲，系统记录 `match_reason=skill+billing`。

「分类对了、技能对了，业务还是错了。」陆沉苦笑，「这类单子其实是『商务+账单』混合意图。」

**第二个产品二选一**更尖锐：

- **方案 C**：混合意图一律不进自动通道，全部人工分诊；
- **方案 D**：加一层「客户价值 × 情绪 × 关键词」规则引擎，命中高风险组合时覆盖模型结论。

方栎和唐玫争论到凌晨。唐玫要 C：「大客户宁可慢，不能错。」方栎坚持 D 加人工兜底：「每天 4,200 条工单，全人工分诊，排班先崩。」

最终妥协：**VIP 且（续费/投诉/情绪高风险）→ 强制升级队列**，自动路由只覆盖长尾标准单。

---

### 第四章：人机分诊台——机器筛，人选最终人

新流程上线后，工单路径变成四层：

**第一层，接入与清洗**：去重、合并同一客户 15 分钟内的重复提交，识别渠道（邮件、IM、工单系统）。

**第二层，意图与客户画像**：模型输出 Top3 意图及置信度；同时拉 CRM 合同金额、续费倒计时、近 30 天投诉次数。

**第三层，规则覆盖**：例如「VIP + 续费关键词 + 负面情绪」直接打标 `escalation_renewal`，无视账单分类；「置信度 < 85% 或 Top2 意图分差 < 10%」标 `triage`。

**第四层，技能路由**：在确定队列后，再按坐席技能、当前负载、SLA 剩余时间选人——不是纯「最闲」，而是「能在 SLA 内响应且技能匹配度最高」。

分诊台界面给值班组长三件事：模型建议、命中了哪条规则、客户 360 摘要。组长可以一键改派，每次改派都记原因，回流训练集。

周四，华创第三条工单进来：「部署失败第三次了，续费款本周能否暂缓支付」。

模型给出 `incident` 68%、`renewal` 71%——几乎打架。规则引擎命中「VIP + 续费 + 技术故障」组合，直接进 `escalation_renewal`。值班组长老周扫了一眼，派给资深客户成功经理何蔚，附带 L2 工程师待命。

何蔚 18 分钟首响，先_ack 部署问题已拉技术，再说明续费条款可单独开会。华创采购总监回复：「这次像回事。」

方栎在监控里看到：这条工单没有走「最快路径」，走了「最对路径」。平均等待时间只多了 4 分钟，投诉升级率却降了 37%。

---

### 第五章：外部知识库挂了，路由不能跟着瞎

周五早高峰，对接的 CRM 接口超时。部分工单拿不到客户等级，意图模型仍正常返回结果。

旧逻辑会默认「非 VIP」走标准队列——等于把风险 silently 降级。

陆沉触发预案：

- CRM 不可用时，工单标 `profile_degraded`，**禁止**进入全自动 VIP 路由；
- 缓存 24 小时内拉过的客户画像兜底，缓存未命中则进分诊台；
- 技术故障类仍可自动派给 L1，但 SLA 缩短一半，逼人工复查；
- 恢复后批量重算「降级期间」的工单，该升级的补升级。

中午 CRM 恢复。复盘会上，唐玫问了一个 PM 该问的问题：「我们省了多少人力？」

方栎摊开账：

- 自动路由覆盖率从 38% 提到 61%，但**不是越高越好**；
- 人工分诊从「全量 4,200 条」变成「1,600 条模糊/高风险」，组长人均多处理 20%，却少接 80% 的投诉升级；
- 误路由导致的销售救火，从每月约 6 次降到 1 次；
- 算法调用 + CRM 查询 + 分诊台人力，折合每张工单成本降 12%，**前提是**把「误路由」算进成本，而不只看 API 账单。

陆沉补了一句：「模型还会漂。下个月客户开始用新黑话骂我们，意图分布又变。路由是运营系统，不是上线即完工。」

方栎写下结论：**智能工单分配，是「意图理解 + 客户画像 + 规则护栏 + 技能匹配 + 可降级分诊」的组合拳；AI 决定建议，人保留否决权。**

那条曾被分给实习生的 380 万工单，华创最终续了。客户说：「你们后面响应像换了团队。」唐玫笑：「团队没变，变的是谁先看到消息。」

# 3. 系统正解

智能工单分配系统的数据流是：**多渠道接入与去重 → 文本/上下文意图分类 → 客户画像与 SLA 策略拉取 → 规则引擎覆盖与风险打标 → 技能匹配与负载均衡 → 分诊台/坐席工作台 → 反馈与路由日志回流**。

- **第一章**对应路由日志与可观测性——没有 `reason` 字段，就无法侦探式排障。
- **第二章**对应意图分类模块与置信度门槛——模型输出不是终局，只是候选。
- **第三章**对应规则引擎——处理混合意图、VIP、情绪等模型不擅长的硬约束。
- **第四章**对应技能路由与分诊台——人在模糊地带的否决权与改派留痕。
- **第五章**对应画像降级与补偿重算——外部依赖失败时，路由策略必须保守，不能默认低风险。

# 4. 类比锚定

**类比：医院急诊预检分诊（Triage）。**

病人进门先分「濒危、紧急、次紧急、非紧急」，再分到对应科室；不是谁先到谁先治，而是**病情 × 资源**匹配。智能工单路由同理：先判「什么事」，再判「多急、多贵」，最后判「谁最能处理」。

| 急诊预检（旧经验） | 智能工单分配（新系统） | 映射关系 |
|---|---|---|
| 主诉与生命体征 | 工单标题、正文、渠道、附件 | 路由的原始信号 |
| 分诊护士判断科室 | 意图分类模型 | 把问题归到候选类型 |
| 既往病史与过敏史 | CRM 客户等级、合同、投诉史 | 同样症状，VIP 路径不同 |
| 红区规则（胸痛直通心内科） | 「VIP+续费+负面情绪」规则覆盖 | 硬规则压过模型软判断 |
| 专科医生排班与当前负载 | 坐席技能矩阵与 SLA 剩余时间 | 不是纯轮询，是匹配+时效 |
| 疑难病例会诊台 | 人工分诊台 + 组长改派 | 机器不确定时的人机协作 |
| HIS 宕机时纸质登记 | CRM 降级缓存 + 禁止自动 VIP 路由 | 依赖挂了，策略要更保守 |

# 5. 类比边界

## A. 系统核心要点

- **输入不止工单正文**：渠道、客户 ID、产品模块、语言、是否重复提交，都会影响路由。
- **模型边界**：意图分类擅长标准表述，对「骂一半谈续费」的混合工单容易偏科；必须输出置信度与 TopN，而不是单一标签。
- **人工介入条件**：低置信、混合意图、VIP 高风险组合、画像降级、客户指定联系人——应走不同队列，不能硬并一条「人工复核」。
- **降级策略**：CRM/意图服务不可用时，宁可多进分诊台，也不能默认非 VIP；恢复后要有补偿重算。
- **成本结构**：不只算模型调用费，要算误路由带来的销售救火、客户流失和组长加班。

## B. 类比失效点

1. **急诊病情相对标准化，工单意图随产品迭代快速变化。** 新功能上线一周，用户说法就换一套，模型需持续回流改派日志。
2. **医院不能挑病人，客服可以设 SLA 差异化。** 同一「技术故障」，免费用户和百万合同客户等待上限不同，路由权重不能一刀切。
3. **急诊流程法规强约束，企业路由策略随商务变。** 大促、续费季、裁员期，「升级」阈值可能要临时调，不能写死。
4. **医患纠纷与客服纠纷的证据链不同。** 路由需保留模型版本、规则命中、改派人与理由，方便复盘，而不是只留最终坐席 ID。
5. **类比掩盖「坐席技能会成长」。** 实习生三个月后可能变高手，技能表若人工月更，路由会系统性低估新人。

# 6. 复盘室

### 复盘 1

- **唐玫说**：「自动分配就是为了省分诊人力，confidence 不够也先派出去再说。」
- **真相是**：客服路由的产出单位是 **「一次正确的首次接触」**，不是「派单速度」。错派给账单专员，省下的 30 秒会变成销售 VP 的两小时灭火。
- **如果你做 PM**：KPI 要同时写 **首响 SLA 达成率** 和 **误路由率/升级率**；只追自动化率，会把高风险单赶进自动通道。

### 复盘 2

- **陆沉说**：「意图模型 91% 准确率，可以全量自动路由了。」
- **真相是**：91% 是 **平均数**，混合意图和 VIP 工单往往在那 9% 里；且错误分布不均——一张 380 万续费单抵一万条密码重置。
- **如果你做 PM**：要求算法按 **客户等级 × 意图类型** 分层报指标；上线前用历史「改派记录」做回放，而不是只看实验室测试集。

### 复盘 3

- **方栎说**：「CRM 挂了就先按模型分，恢复了再手工改。」
- **真相是**：画像缺失时继续自动 VIP 路由，等于 **在盲人状态下开快车**；降级应默认「多人工、少自动」，并触发告警值班。
- **如果你做 PM**：给每个外部依赖画 **失败状态机**——超时多久、缓存多久、禁止哪些自动路径、恢复后如何批量补偿。

# 7. 学习资料

1. **[Zendesk：智能路由与技能匹配产品文档](https://support.zendesk.com/hc/en-us/articles/4408883597850-Setting-up-skills-based-routing)**  
   **适合**：第一次做客服路由 PRD 的 PM。  
   **解决什么**：理解技能表、优先级、溢出队列在成熟工单系统里长什么样，避免从零发明名词。

2. **[Google Cloud Dialogflow：意图（Intents）与置信度阈值](https://cloud.google.com/dialogflow/docs/intents-overview)**  
   **适合**：需要和算法对齐「意图分类」边界的 PM。  
   **解决什么**：搞清 Top intent、fallback、置信度门槛如何映射到「自动 vs 人工」产品策略。

3. **[ITIL 4：Incident Management 与优先级矩阵](https://www.axelos.com/certifications/itil-service-management/itil-4-foundation)**  
   **适合**：做 ToB 客服、要和客户签 SLA 的 PM。  
   **解决什么**：把「紧急程度 × 影响范围」写成可路由的优先级，而不是凭感觉标 P0/P1。

4. **[Intercom 博客：When to use humans in the loop for AI support](https://www.intercom.com/blog/ai-customer-service/)**  
   **适合**：权衡自动化率与服务质量的 PM。  
   **解决什么**：行业案例说明「全自动」与「AI 建议+人确认」在不同客群下的 ROI 差异。
