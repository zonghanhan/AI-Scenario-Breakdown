# 2. 业务场景故事

## 《暴雨夜的 47 单，派单算法挨了三次骂》

### 第一章：苏州在下雨，工单在排队

晚上 7 点 14 分，「安修到家」华东调度中心的大屏从绿色变成橙色。

台风外围带来的暴雨横扫苏州工业园区。过去 40 分钟，客服热线涌进 47 个紧急报修单：漏水、漏电、空调外机异响、燃气热水器点不着火。其中 12 单标注了「老人独居」「婴幼儿在家」「漏水已淹到插座」——这类单子在系统里叫 **P0 紧急单**。

值班调度主管孙姐盯着待派工队列，手指已经放在「人工全接管」按钮上。她干了八年家电售后，知道暴雨夜的规律：**不是没人接单，是单子会扎堆、路会堵、师傅会拒、客户会骂**。

刚转岗来做 AI 产品的赵航坐在旁边，盯着上周刚上线的「智能派单 2.0」仪表盘，心里还有点小骄傲：「孙姐，让算法先跑一版吧。我们加了地理位置、技能标签、实时路况，比人工派快多了。」

孙姐没抬头：「快是快，但客户要的不是『派得快』，是『师傅真的能在承诺时间到』。你算法上次把李师傅从吴中派到相城，导航显示 28 公里、52 分钟——客户页面写的是『预计 30 分钟内上门』。」

运营总监王总从会议室探出头：「今晚目标很简单：P0 单 2 小时内上门率不低于 95%，普通单 4 小时内不低于 80%。投诉不能超过 3 单。赵航，你盯着系统；孙姐，你兜底。开始。」

**悬念**：47 个单子、23 个在途师傅、暴雨红色预警——AI 的第一版派单方案，能撑过这个夜晚吗？

---

### 第二章：「距离最近」的陷阱

7 点 18 分，赵航点击「智能派单」。系统后台开始运转——不过此刻在调度台看来，只是一次批量刷新。

对每个待派工单，系统做了几件事：

1. **解析工单需求**：从报修描述里抽出设备类型（挂机空调 / 燃气热水器 / 电路）、故障现象（漏水 / 异响 / 无法点火）、所需技能等级（普通 / 高级 / 燃气资质）。
2. **圈定候选师傅池**：只保留「当前在岗 + 技能匹配 + 服务半径内 + 未达当日接单上限」的师傅。
3. **计算综合得分**：直线距离 40%、实时路况 ETA 30%、师傅历史同品类完工率 20%、当前负载 10%。
4. **输出推荐**：每个工单绑定一名师傅、一个预计到达时间、一条路线。

第一版结果 38 秒出炉。孙姐扫了一眼，脸色就变了。

工单 #20260719-0031：工业园区某小区，燃气热水器漏水，P0。系统派给了 **李师傅**——他此刻在吴中区装空调，直线距离 4.2 公里，全场最近。

但孙姐一眼看出问题：「李师傅没有燃气资质。他空调装得好，热水器不能碰。」

赵航愣了：「技能标签里他勾了『热水器维修』啊。」

孙姐冷笑：「那是他三年前学的，去年资质过期了，系统没同步。这种单子派出去，出了事谁担？」

第二处异常更明显：工单 #20260719-0044，相城区漏水 P0，派给了 **周师傅**——距离最近，ETA 显示 22 分钟。但孙姐知道周师傅今晚已经接了 6 单，上一单还在客户家拆洗衣机，至少还要 40 分钟才能出门。

赵航打开师傅端 App 一看：周师傅状态仍是「空闲」——因为他忘了点「完工」，GPS 显示他停在上一单小区门口已经 25 分钟了。

7 点 25 分，王总走过来：「第一版派了 31 单，客户侧已经弹出 9 个『预计到达时间』。有 4 个 ETA 明显不靠谱。赵航，**在客户看到错误承诺之前**，全部撤回重派。」

赵航第一次意识到：工单分配（分给谁）和工程派单（这人什么时候能到）之间，隔着一整条物理世界。

**悬念**：修完「距离最近」的坑，下一刀该砍向哪里？

### 第三章：时间窗口，比距离难十倍

7 点 35 分，赵航带着研发紧急打了三个补丁：

- **硬约束过滤**：燃气类工单必须校验「燃气资质有效期」，过期直接剔除候选池，不靠标签。
- **状态机校正**：师傅 GPS 静止超过 15 分钟且上一单未完结，自动标为「忙碌」，不进入空闲池。
- **ETA 重算**：用第三方路况 API 替代直线距离估算，并加 15 分钟「收尾缓冲」——师傅总得洗手、收拾工具、下楼吧。

第二版派单出炉。孙姐这次没骂人，但提出了更刁钻的问题。

工单 #20260719-0052：客户备注「**明天早上 8 点前要出门，请务必今晚修好**」。地址在姑苏区老城巷子里，晚上 8 点后货车禁行。

系统把单子派给了技术最好的 **陈师傅**，ETA 21:40。看起来合理。

孙姐却摇头：「老城巷子晚上 7 点半以后进不去厢式货车。陈师傅开的是面包车，能进，但他今晚已经排了 4 单，按历史数据他平均每单 55 分钟——你这 21:40 是做梦。」

赵航调出陈师傅的 **时间窗日历**：系统只知道他「有空」，不知道他 20:30 之后不接单；更不知道这条巷子在暴雨天积水，步行还要扛 20 公斤工具箱走 200 米。

另一个单子更有代表性。客户选了「**预约明天上午 9:00-11:00 安装**」——这是安装单，不是紧急维修。但系统因为「距离最近」，把单子提前塞进了今晚的派单队列，师傅端弹出通知，客户手机也收到「师傅将在 40 分钟内到达」。

客户打电话骂客服：「我明早才在家，你们今晚来干嘛？」

赵航在白板上写下两个词：**硬时间窗**（客户明确说几点到几点）和 **软 ETA**（系统估算的到达时间）。他把两者混在一个字段里展示了——这是产品事故，不是算法事故。

7 点 50 分，团队达成共识：

- **紧急维修单**：以 ETA 为主，但必须叠加路况、技能、资质、师傅真实负载；
- **预约安装/巡检单**：以用户选定的时间窗为主，只在窗内找最优师傅，**绝不为了「近」而打破预约**；
- **所有对外承诺时间** = 系统 ETA + 人工可调缓冲 + 上限封顶（P0 最长不超过 120 分钟）。

**悬念**：规则修完了，师傅本人不买账怎么办？

### 第四章：师傅拒单，算法没有「面子」

8 点 12 分，第二版派单发出 27 分钟后，师傅端开始弹「拒单」。

**张师傅**拒了工单 #20260719-0067：「这个小区我去年被投诉过，客户难缠，我不去。」

**王师傅**拒了 #20260719-0071：「单子写的是挂机空调异响，备注里有一句『可能漏氟』——漏氟要焊补，我工具没带，去了也白跑。」

**小刘师傅**没拒单，但打了调度电话：「系统派我连续跑 5 单，最后一单在太湖边，回程要过跨湖大桥——暴雨封路了，我现在过不去。」

孙姐一边接电话一边手动改派，嘴里念叨：「算法只会算分，不会算『这人愿不愿意去』、『去了能不能一次修好』。」

赵航起初觉得师傅在找茬：「拒单率 18%，是不是管理问题？」

孙姐给他看了一张内部表——**一次上门解决率（FTF, First-Time Fix）**：

| 师傅 | 本月 FTF | 拒单率 | 客户评分 |
|---|---|---|---|
| 张师傅 | 91% | 12% | 4.8 |
| 周师傅 | 62% | 3% | 4.1 |

「张师傅拒单多，是因为他知道自己搞不定的活不接，去了反而差评。周师傅从不拒单，但经常要跑第二趟。」

这句话点醒了赵航。派单系统的目标函数不应该是「派出去多少单」，而应该是 **「一次修好 + 按时到达 + 客户满意」**——有时候，让好师傅拒掉不靠谱的单子，整体效率更高。

8 点 30 分，产品方案调整：

1. **拒单原因结构化**：工具不足 / 技能不匹配 / 路况不可达 / 客户历史纠纷 / 其他（需备注）。
2. **拒单后自动回退**：系统在 60 秒内扩大搜索半径，找下一候选；同时把拒单原因反哺给工单——「可能漏氟」自动升级技能需求。
3. **人工 override 保留**：孙姐可以一键「锁定师傅」或「踢出师傅」，任何 AI 派单结果都可被覆盖，且记录原因用于训练。
4. **连环单上限**：暴雨模式下，同一师傅连续派单不超过 3 单，且总驾驶时间不超过 90 分钟。

王师傅拒单 #0071 后，系统把「漏氟」关键词识别出来，改派给有焊补资质的 **老周**，并备注「请携带焊具」。老周 9 点 05 分一次修好，客户给了五星。

孙姐难得点头：「这单 AI 总算帮了忙。」

**悬念**：夜里 11 点，系统突然告警——路况 API 限流了。

---

### 第五章：API 挂了，承诺不能挂

10 点 47 分，第三方路况服务触发 QPS 限流。47 个工单高峰期过去后，仍有 11 单在途、6 单待派。系统的 ETA 全部显示「计算中」，客户 App 上的倒计时停在原地。

旧方案很简单：限流就无限重试，或者回退到直线距离——赵航在白天演示时觉得「差不多就行」。

孙姐直接否了：「直线距离在暴雨夜误差能到 40 分钟。客户看着倒计时不走，会以为我们跑路了。」

赵航翻出降级清单——这次是他主动写的：

| 级别 | 触发条件 | 行为 |
|---|---|---|
| L0 正常 | 路况 API 可用 | 实时 ETA + 动态改派 |
| L1 降级 | API 延迟 > 3s 或错误率 > 10% | 切备用路况源；仍失败则用「历史同时段均值 ETA」 |
| L2 降级 | 全部路况源不可用 | 暂停自动改派；已派单展示「时间可能波动，调度员确认中」；新单走人工 |
| L3 兜底 | 系统整体异常 | 孙姐全接管；App 停止推送新 ETA，改为短信通知 |

10 点 52 分，系统进入 L1。备用源撑了 8 分钟后也变慢，自动降到 L2。客户 App 文案从「预计 23:15 到达」变成「已安排师傅，因天气原因到达时间可能调整，调度员将短信确认」——**主动降低承诺精度，好过给出错误承诺**。

孙姐手动确认了 6 个待派单，其中 2 个因为跨湖大桥封闭改派到南岸师傅。凌晨 0 点 17 分，最后一个 P0 单完结。

---

### 第六章：清晨复盘，三次骂换来一张表

第二天早上 9 点，王总主持复盘。大屏上挂着昨晚的核心指标：

- P0 单 2 小时内上门率：**91.7%**（11/12，差 1 单因跨湖封路）
- 普通单 4 小时内上门率：**83.3%**
- 客户投诉：**2 单**（均因首次 ETA 偏差，已在 L2 降级后平息）
- 一次上门解决率：**78.4%**
- 师傅拒单后自动改派成功率：**94.1%**
- 智能派单占比：**67%**（其余 33% 为人工新建或改派）

赵航以为会被骂惨。王总却先说：「三次骂，骂对了。」

- **第一次骂**：「最近 ≠ 最快 ≠ 最能修」——距离只是派单特征之一。
- **第二次骂**：「预约单和紧急单不能混排」——时间窗是硬约束，不是展示字段。
- **第三次骂**：「师傅拒单不是捣乱」——拒单原因是宝贵的标注数据。

孙姐把复盘总结成一张给产品经理的表：

> **工程派单 ≠ 地图找最近的人。它是「带硬约束的实时组合优化」+「可解释的人工兜底」+「对外承诺的分级管理」。**

赵航在文档里写下今晚最重要的认知：智能工单分配决定「这事归谁管」，工程派单决定「这个人什么时候以什么状态出现在什么地点」——后者多了一整个物理世界，也多了一整层「承诺管理」。

窗外雨停了。孙姐泡了杯茶：「下次别跟我说『算法比人快』。暴雨夜，**快派错单，比慢派单更贵**。」

# 3. 系统正解

工程类 AI 派单系统的核心数据流是：**工单接入与需求解析 → 技能/资质/设备匹配 → 师傅状态与位置实时同步 → 路况与时间窗约束下的路径规划 → 综合评分派单 → 师傅确认/拒单 → 在途监控与动态改派 → 完工回写与 SLA 结算**。

- **第一、二章**对应工单解析与候选池过滤——「燃气资质过期」「师傅状态未更新」说明 **AI 只能信经过校验的结构化数据**，不能盲信标签。
- **第三章**对应时间窗引擎与对外承诺层——预约安装和紧急维修是两条轨道，ETA 不是给用户看的装饰。
- **第四章**对应拒单回流与人工 override——师傅是活人，系统必须留出「不执行」的通道并把原因结构化。
- **第五、六章**对应降级策略与 SLA 复盘——路况 API 挂了时，降级的是「承诺精度」，不是「服务中断」。

**AI 产品岗的典型应用场景**：

- **需求文档**：写 PRD 时把「派单」拆成硬约束（资质、时间窗、服务半径）和软优化（距离、负载、FTF），禁止混成一个「智能推荐分」。
- **方案评审**：问研发「ETA 置信区间多少」「拒单后回退 SLA 是多少秒」「人工 override 是否全量留痕」——这三个问题能筛掉一半不靠谱方案。
- **指标设计**：主指标用「一次修好率 + 准时到达率」，辅指标用「自动派单占比」；后者高不等于系统好。
- **跨团队沟通**：跟算法说「目标函数要包含 FTF，不是纯距离」；跟运营说「拒单原因是标注资产，别当纪律问题」；跟客服说「降级时统一话术，别让客户看到卡死的倒计时」。

# 4. 类比锚定

**类比：网约车调度中心。**

乘客下单 ≈ 客户报修；司机空驶状态 ≈ 师傅空闲；导航 ETA ≈ 预计到达时间；拼车顺路 ≈ 连环派单；高峰溢价 ≈ P0 紧急单优先调度；乘客取消 ≈ 师傅拒单。PM 对「派单—接单—到达—评价」闭环不陌生，差别在于工程派单多了「技能匹配」和「一次修好」的硬约束。

| 网约车调度（旧经验） | 工程 AI 派单（新系统） | 映射关系 |
|---|---|---|
| 乘客下单与起终点 | 报修/安装工单 + 地址 + 时间要求 | 定义派单需求的输入 |
| 车型匹配（快车/专车/六座） | 技能标签 + 资质证书 + 工具要求 | 不是所有人都接得了所有单 |
| 司机空驶 / 载客中状态 | 师傅空闲 / 在途 / 作业中 / 离线 | 状态错误会导致「假空闲」 |
| 导航 ETA 与路况 | 第三方路况 + 历史均值 + 缓冲时间 | 对外承诺的时间来源 |
| 乘客预约上车时间 | 客户选定安装/巡检时间窗 | 硬约束，不可为「近」而打破 |
| 司机拒单 / 乘客取消 | 师傅拒单 + 结构化原因 | 回退重新派，原因反哺需求 |
| 人工调度台 override | 孙姐一键改派 / 锁定 | AI 建议，人可否决且留痕 |
| 行程结束乘客评分 | 完工回写 + FTF + 客户评价 | 闭环数据用于优化下一轮派单 |

# 5. 类比边界与系统要点

## A. 系统核心要点总结

- **数据输入要求与格式规范**：工单需结构化设备类型、故障现象、紧急等级、时间窗类型（立即/预约）；师傅侧需实时 GPS、技能资质有效期、工具清单、当日负载——**过期数据比没有数据更危险**。
- **AI 模型的能力边界与置信度**：路线规划、需求解析（NLP 抽故障关键词）、到达时间预测都有置信度；低置信时应扩大候选池或转人工，不能把 ETA 伪装成精确值。
- **人工介入的触发条件与流程**：资质缺失、时间窗冲突、连环单超限、路况异常、客户历史纠纷、拒单超过 N 次——均应触发人工调度台；override 必须留痕供审计和训练。
- **系统的容错与降级机制**：路况 API 限流时降级到历史 ETA + 话术降级；核心服务不可用时人工全接管；**已发出的错误承诺要及时撤回或主动更正**。
- **成本结构**：算法计算成本通常不高；真正的大头是 **空驶里程、二次上门、超时赔付、客户投诉处理、师傅流失**——派错一单的成本远高于多算几秒。

## B. 类比失效点

1. **网约车拉客「到了就行」，工程服务要「一次修好」。** 网约车把乘客送到目的地即完成任务；师傅到了却发现缺工具、缺资质，第二趟的成本不会出现在导航 ETA 里。
2. **网约车司机供给相对同质化，师傅技能高度异构。** 同一距离下，能修燃气的可能只有两人——这不是「调更多车」能解决的。
3. **网约车 ETA 偏差几分钟用户能忍，漏水淹到插座差 30 分钟是安全事件。** 紧急等级的业务后果不对称，不能套用同一套容忍区间。
4. **网约车有动态加价平衡供需，家政/维修行业不能简单照搬。** 暴雨夜逼师傅出门涉及劳安、保险、劳动关系，产品策略要过法务和运营红线。
5. **类比掩盖「拒单」的价值。** 网约车司机挑单常被负面看待；工程场景里，好师傅的合理拒单是质量信号，系统应学习而非惩罚。

# 6. 深度业务洞察

### 洞察 1：客户要的是「确定感」，不是「智能感」

- **表面需求**：「能不能自动派单，省调度人力？」
- **真实需求**：「我约了 9-11 点，师傅能不能 9 点前到？」「漏水了，能不能告诉我一个靠谱的时间，别一会儿 20 分钟一会儿 40 分钟？」
- **产品坑**：把内部「推荐师傅」直接展示成对客户的「承诺到达」——一旦算法改派，客户信任崩塌。对外应用「确认后承诺」或「区间承诺」，内部再用精确 ETA 调度。

### 洞察 2：师傅端体验决定系统上限

- **表面需求**：「把单派出去就好。」
- **真实需求**：师傅需要看清「为什么派给我」「上一单几点能完」「这单赚不赚」「拒了会不会扣钱」。
- **技术坑**：只优化客户端 SLA，不把拒单原因、路程补偿、连环单间隔做到师傅端可理解——拒单率会隐性升高，数据却显示「派单成功」。

### 洞察 3：状态机比算法模型更先出问题

- **业务侧**：80% 的「派单不准」投诉，根因是师傅状态未更新、资质未同步、地址 geocoding（地址转经纬度）偏差——不是排序模型不够好。
- **工程坑**：上线炫目的「AI 派单」前，先审计主数据：资质谁维护？状态谁更新？地址谁校验？否则模型只是在错误的世界上做精确计算。

### 洞察 4：天气和路况是「产品开关」，不是「技术细节」

- **运营侧**：暴雨模式应预先定义——连环单上限、服务半径收缩、对外话术切换、赔付规则激活。等 API 限流了再开会，一定来不及。
- **合规侧**：强制师傅暴雨出远单，可能触及劳动保护和保险条款；产品需有「安全拒单」白名单，别只靠算法硬推。

### 洞察 5：ROI 要算「少一次二次上门」，不是「少一个调度员」

- **商业侧**：卖方案时客户爱听「调度人力减半」；续费时看的是「一次修好率提升 5 个点，二次上门成本降多少」。
- **定价坑**：按「派单次数」收 SaaS 费会激励客户刷自动派单——应绑定 SLA 达成、FTF、客户满意度等业务结果指标。

### 官方文档 / API 参考

1. **[高德开放平台：路径规划 API（驾车 ETA）](https://lbs.amap.com/api/webservice/guide/api/direction)**

   **适合**：需和算法同学对齐「组合优化」边界的产品经理。  
   **解决什么**：用白话搞懂「为什么 47 单不能贪心最近就完事」；不必会写代码，要知道 VRP 是什么级别的难题。

### 行业最佳实践

3. **[Salesforce Field Service：Scheduling 官方文档](https://help.salesforce.com/s/articleView?id=sf.fs_scheduling.htm)**  
   **适合**：做 B2B 现场服务、维保派单的 PM。  
   **解决什么**：看成熟产品如何把技能、时间窗、移动劳动力管理组合在一起——中企可对照「我们缺了哪块」。

   **适合**：有互联网背景、想借鉴即时配送经验的 PM。  
   **解决什么**：理解大规模实时调度里「预测」「改派」「峰值降级」的通用套路；注意工程派单多了「技能异构」约束。

5. **[Gartner 术语：Field Service Management (FSM)](https://www.gartner.com/en/information-technology/glossary/field-service-management-fsm)**  
   **适合**：需要向老板或客户解释赛道的 PM。  
   **解决什么**：把「派单系统」放进 FSM 品类，说清楚和 CRM、工单系统、资产管理的关系。

   **解决什么**：30 分钟后你应该能答出孙姐昨晚三个问题的产品解法——比看十篇概念文更管用。
