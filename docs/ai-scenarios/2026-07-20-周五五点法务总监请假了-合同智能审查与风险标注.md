# 2. 业务场景故事

## 《周五五点，法务总监请假了》

### 第一章：68 页 PDF 和一条「周一必须签」

周五 17:02，SaaS 公司「云栈」的采购总监韩牧把一份合同丢进飞书群：「数据中心三年托管，总价 318 万，供应商今晚六点前等签字版。周一董事会要过，不能再拖。」

附件是《云栈-启明 IDC 托管服务主协议》加三份附件，共 68 页。排版是典型的律所模板：罗马数字章节、交叉引用、定义条款塞在附录里，还有一份单独的《服务等级协议（SLA，就是承诺可用性和赔偿标准的那份附属协议）》。

法务总监周澜两小时前因急性肠胃炎去了医院。值班法务实习生小柯刚入职三周，看到群消息手都在抖：「这种量级……我最多能核对签章页和金额。」

韩牧 @ 了刚转岗来做 AI 产品的林晚：「你们上半年立项的合同智能审查，现在能顶多少？」

林晚心里一沉。系统上个月刚灰度，只覆盖标准采购模板，还没碰过 IDC 这种重资产长周期合同。但她还是打开后台看了一眼：「能跑。但今晚出来的只能是**风险标注稿**，不是法律意见。签字决定仍然要过人。」

**悬念**：林晚答应的「能跑」，到底是救命稻草，还是给周一埋雷？

---

### 第二章：上传之后，系统做的第一件事不是「找风险」

林晚把 PDF 拖进审查台。界面没有立刻刷出一堆红字，而是先走**文档接入层**——这一步和发票 OCR 的「收件检查」很像：

- 文件是否可解析：扫描件还是可选中文字的数字 PDF？
- 有没有密码、缺页、页码断裂；
- 是否混入了报价单、营业执照等非合同正文；
- 合同类型路由：采购 / 销售 / 劳动 / NDA（保密协议）走不同审查模板。

68 页里，前 4 页是封面和签署页，正文从「第一条 定义」开始。系统识别出这是**采购类服务合同**，自动挂上「云栈采购合同审查清单 v2.3」——里面预置了财务、信息安全、法务三方各自关心的检查项。

接着是**条款切分（Clause Segmentation）**：不是按页切，而是按「第 X 条」「第 X.Y 款」切成 214 个条款块，并建立引用关系——比如正文里写的「按附件二 SLA 执行」，会链接到附件二对应段落。

小柯凑过来看：「所以它已经知道『第三条』和『附件二第四条』是一伙的？」

林晚点头：「不懂结构，后面的大模型（用大段上下文理解文字的 AI）只能瞎猜。切分错了，风险就会标错地方——这比不标更危险。」

三分钟后，状态从「解析中」变成「待审查」。真正的 AI 环节，这时才开始。

---

### 第三章：六个红灯亮起，但有一个是误报

系统并行跑三条线：

1. **条款抽取**：把付款节点、违约金比例、知识产权归属、数据出境、争议管辖等抽成结构化字段；
2. **风险识别**：用「规则 + 模型」对照公司红线与历史纠纷案例，给每条打风险等级；
3. **偏离度比对**：和云栈标准采购模板逐条 diff，标出「比模板更利于对方」的改写。

屏幕上弹出 **6 个红灯、11 个黄灯、43 个绿灯**。韩牧冲进来：「红的什么？」

林晚逐条讲：

| 红灯 | 系统判断 | 条款位置 |
|------|----------|----------|
| 1 | 单方解除权不对等：对方可随时终止，我方需提前 90 天 | 第 12.4 条 |
| 2 | 赔偿责任上限仅「直接损失」，且封顶为合同总额 10% | 第 15.2 条 |
| 3 | 数据托管地点未明确境内，默认可能出境 | 第 8.1 条 |
| 4 | SLA 可用性 99.5%，低于公司 IDC 类最低要求 99.9% | 附件二 2.1 |
| 5 | 自动续约 + 涨价条款：续约时对方单方调价 | 第 3.6 条 |
| 6 | **争议管辖：对方所在地法院** | 第 18.2 条 |

韩牧脸色发白：「第六条绝对不能签。」

然而小柯点开第六条，愣了：正文写的是「提交上海国际经济贸易仲裁委员会仲裁」，附件签署页却印着「杭州市西湖区人民法院」。系统把**附件签署页的印刷错误**当成了正文管辖条款，标了红灯。

林晚在工单里点「误报反馈」，并手动把该条降级为「文本不一致，需澄清」。她在心里记了一笔：**合同审查的误报，代价是信任崩塌——业务方以后会把所有红灯当狼来了。**

另外，黄灯里有一条「保密期限 5 年」，公司模板是 3 年。韩牧说：「IDC 行业都 5 年，这条我能接受。」林晚把公司政策里的「行业例外白名单」勾上，黄灯变绿——**风险不是绝对真理，必须接业务政策引擎。**

**悬念**：六个红灯修完五个，剩下的「赔偿责任上限 10%」谈不拢，韩牧说：「那我把总价砍 30 万，风险是不是就对等了？」

---

### 第四章：AI 不会帮你谈，但能告诉你「改了什么」

夜里 9 点，供应商法务回了一版修订稿。韩牧问：「能不能再跑一遍，看对方改没改？」

林晚启动**版本对比审查**。系统不是简单 diff 文字，而是：

- 按条款 ID 对齐新旧版本；
- 重新计算风险分：解除权仍不对等，但赔偿上限从 10% 提到 30%；
- 标出**隐性变更**——对方把「数据不得出境」从正文删了，塞进了附件三的「技术实施说明」里，字体更小、位置更偏。

小柯差点漏看附件三。系统在侧栏提示：「第 8.1 条数据位置承诺已删除，附件三第 7 段新增『经客户书面同意可跨境传输』，语义弱于标准模板。」

这就是条款抽取 + 风险识别的组合价值：**人读 68 页容易疲劳，机器擅长「同一主题跨附件追踪」**。

但谈到赔偿上限，韩牧想用「降价换条款」——林晚调出了公司的**风险量化表**（不是精确法律计算，而是产品侧的可比维度）：

- 托管业务过去三年因宕机产生的直接索赔均值：约合同额的 18%；
- 10% 上限意味着极端情况下公司自担超额部分；
- 降价 30 万并不自动等价为「风险下降」，因为 SLA 未同步提高。

她把结论写在批注里：**「建议谈判包：赔偿上限 ≥25% + SLA 提到 99.9% + 数据境内明确写死，降价可作为筹码但不应单独置换。」**

周澜在医院语音里回了一句：「AI 标的方向对，签字前我视频过一遍。别让它替你做谈判策略。」

**悬念**：周一董事会前，还有最后一关——董事会要的不是「红灯清单」，而是一张能讲清楚「为什么还能签」的风险接受说明。

---

### 第五章：绿灯不等于能签，董事会要的是「谁背锅」

周日晚上，林晚导出《合同审查报告 v3》：6 页摘要 + 214 条条款索引 + 5 项未完全关闭的红线及**风险接受理由**。

其中两项经谈判关闭：管辖改仲裁、数据境内写入正文。三项以「有条件接受」进入董事会：

1. 赔偿上限 25%（未达模板 50%，但高于初稿 10%）；
2. SLA 99.7%（折中）；
3. 自动续约需我方提前 60 天书面反对，否则视为同意。

系统在报告末尾自动生成了**审计轨迹**：谁上传、哪版模型、哪条规则命中、谁点了误报、谁批准风险接受——这对上市公司合规审计不是锦上添花，是刚需。

周一 10 点，董事问：「AI 审的，出了问题算谁的？」

林晚答：「算流程。AI 负责**召回可疑条款**，法务负责**定性**，业务负责**议价**，系统记录**谁批准例外**。没有『模型说能签』这个按钮。」

11 点，合同签署。韩牧松口气：「要是没有周五那版标注，小柯估计要通宵，我也看不出附件三那个坑。」

周澜事后复盘：「这套东西最大的价值，不是替法务省 6 小时，而是**把『法务看过』从黑盒变成可检索、可统计、可迭代的条款知识库**——下次 IDC 合同再来，红灯规则能更准。」

林晚在文档里写下定义：**合同智能审查与风险标注系统 = 文档解析与条款切分 + 结构化抽取 + 规则/模型风险识别 + 模板偏离比对 + 人机协同复核台 + 版本 diff + 审计与知识沉淀**。不是「AI 律师」，是「法务和采购的共同工作台」。

# 3. 系统正解

合同智能审查系统的核心数据流是：**合同文档接入与类型路由 → OCR/解析与条款切分 → 关键条款结构化抽取 → 规则引擎 + 风险模型并行标注 → 与公司标准模板/红线库比对 → 风险分级与工单分流 → 法务/业务复核台 → 谈判版本 diff 再审查 → 审查报告与风险接受审批 → 条款知识库回写**。

- **第一章**对应业务触发与权限边界——紧急签约场景下，系统输出的是「标注稿」而非法律结论。
- **第二章**对应文档解析与条款切分——结构错了，后面全错；这是比「调大模型」更基础的能力。
- **第三章**对应风险识别与政策引擎——红灯要结合公司红线、行业例外和误报反馈，不能只有模型一张嘴。
- **第四章**对应版本对比与跨附件语义追踪——谈判场景的高频刚需，也是纯人工最易漏项之处。
- **第五章**对应审计、风险接受工作流和知识沉淀——让「这次拍板」成为「下次更准」的训练信号。

**AI 产品岗典型应用场景：**

| 场景 | 具体例子 |
|------|----------|
| 需求文档 | 写清「支持 PDF/Word、扫描件 OCR、条款级而非页级标注、与标准模板 diff、风险接受需双人审批」；明确**不做**自动法律结论与自动签署 |
| 方案评审 | 争论「召回率 95% 够不够」时，要求同步定义**误报率**和法务复核工时——高召回 + 高误报 = 信任破产 |
| 指标设计 | 核心指标：高风险条款召回率、误报率、单份合同法务节省时长、**谈判轮次减少数**、同类合同审查周期；北极星是「可关闭风险项」而非「标注条数」 |
| 跨团队沟通 | 对法务说「模型帮你找疑点」；对业务说「红灯不是否决，是谈判清单」；对工程说「条款 ID 对齐比全文 diff 重要」 |

# 4. 类比锚定

**类比：新房交付前的专业验房。**

买房人请验房师带着 checklist 上门：量尺寸、查空鼓、测漏水、看电路——不是替你做「买不买」的决定，而是把**隐蔽工程问题标在图上**，方便你和开发商谈整改。合同审查 AI 就是「条款验房师」：标准清单来自公司历史教训（验房国标 + 小区常见坑），标注的是「哪里可能漏水」，不是「这房值不值」。

| 验房（旧经验） | 合同智能审查（新系统） | 映射关系 |
|----------------|------------------------|----------|
| 户型图与施工图纸 | 合同正文 + 附件 + 交叉引用关系 | 没有图就查不清哪堵墙是承重墙 |
| 验房 checklist（水电、防水、门窗） | 公司红线库 + 合同类型审查清单 | 把隐性经验变成可执行检查项 |
| 空鼓锤、测距仪、相位仪 | OCR、条款切分、NER 抽取、风险模型 | 不同工具有不同精度，不能一把锤子验所有 |
| 验房报告上的红/黄标贴纸 | 风险分级标注（高/中/低） | 可视化优先级，不是非黑即白 |
| 开发商返修后二次验收 | 谈判版本 diff 再审查 | 整改是否到位，要比对条款而非只看封面 |
| 业主最终决定收房 | 法务 + 业务风险接受审批 | 工具不提供「自动收房」按钮 |
| 小区业主群里的「常见坑」沉淀 | 条款知识库与案例库 | 这次漏水修好了，下次别的小区能预警 |

# 5. 类比边界与系统要点

## A. 系统核心要点

- **数据输入要求与格式规范**：需支持可编辑 PDF、扫描件、Word；要识别合同类型并路由模板；元数据（我方主体、对方主体、金额、期限）影响审查规则；混排附件、多版本签署页是常见脏数据源。
- **AI 模型的能力边界与置信度**：模型擅长语义理解和跨段归纳，但不承担法律责任；对数字（违约金比例、期限）、管辖机构名称等需规则校验；低置信标注应默认升高人工复核优先级。
- **人工介入的触发条件与流程**：新合同类型、首份供应商、红线命中、模型低置信、业务申请例外接受、金额超阈值——应走不同审批链；实习生只能改标注，不能点「风险接受」。
- **系统的容错与降级机制**：解析失败时退回「纯人工 + 基础 OCR」，不输出假绿灯；主模型不可用时规则引擎仍可跑硬性红线；版本 diff 失败时保留全文对比兜底。
- **成本结构**：计算成本（长文档多轮 LLM 调用）随页数线性涨；法务复核时间是隐藏大头；**误报导致的信任损失**和**漏报导致的诉讼成本**远超 API 账单；知识库维护需要法务持续标注。

## B. 类比失效点

1. **验房标准相对客观，合同风险高度依赖商业立场。** 同样「赔偿上限 25%」，战略供应商和一次性采购，业务容忍度完全不同——系统必须接「政策引擎」，不能只有对错。
2. **房子不会一夜改结构，合同谈判版一晚上能改十轮。** 版本对齐和隐性挪条款（正文删、附件加）是验房类比覆盖不了的动态博弈。
3. **验房漏项通常可补救，关键条款漏审可能锁死三年。** 合同审查的「漏报成本」不对称，产品上要偏向**高召回**，再用流程消化误报。
4. **AI 输出是概率性的。** 同一条款两次标注可能因模型版本、上下文窗口切分不同而波动——需要规则锚定关键字段，并保留完整审计日志，不能像验房仪一样默认「读数绝对稳定」。
5. **验房师不承担法律责任，企业里「谁批的风险接受」要可追溯。** 类比掩盖了合规、上市审计、内控签名等组织设计，这些是 PM 必须画进流程图的。

# 6. 深度业务洞察

### 洞察 1：业务要的是「谈判弹药」，不是「法律论文」

韩牧周五最怕的不是看不懂法律术语，而是周一董事会问「风险多大、还能不能签」时没话讲。表面需求是「自动审合同」，真实需求是**可执行的谈判清单 + 风险接受话术 + 版本前后对比**。若报告只有「高风险」标签没有「建议改法、_fallback 条款、行业惯例」，法务省下的时间会被业务骂回来。

### 洞察 2：条款切分质量决定天花板，大模型救不了烂 PDF

很多项目把预算全砸在「最强 LLM」上，却忽视 OCR 和条款边界识别。扫描件里「第 15 条」被识别成「第 l5 条」，后面所有风险都标错位——业务一次误报就会永久不信系统。**工程化难点在文档理解流水线，不在 prompt 写得花。**

### 洞察 3：误报和漏报的代价不对称，指标不能只看召回率

漏掉「数据出境」可能引发合规事故；把仲裁条款误标成法院管辖，会让业务忽视真正红线。产品上要同时监控**误报率**和法务「一键驳回」率，并建立误报反哺规则库的机制——否则模型越升级，业务越绕开系统。

### 洞察 4：冷启动靠「标准模板 diff」，飞轮转靠「历史谈判结局」

第一份 IDC 合同几乎没有行业样本，最有效的是和公司采购模板做偏离比对。签完、谈完、纠纷发生后，要把「最终签署条款 + 是否出事」回写知识库——否则 AI 永远停在「会标红」，不会学会「这条黄灯在云栈其实常接受」。

### 洞察 5：合规与商业：审计轨迹不是可选功能

上市或融资中的公司，董事会会问「谁批准了例外」。审查系统必须从 Day 1 设计**不可篡改的操作日志**：哪版文档、哪条规则、谁降级、谁签风险接受。缺这个，法务宁可不用 AI，因为工具会变成合规负债。

   **解决什么**：如何把付款期限、违约金等抽成稳定 JSON 字段，而不是让模型自由发挥写散文。

### 行业最佳实践

3. **[Kira Systems 官方博客与案例（被 Litera 收购的合同分析产品）](https://kirasystems.com/resources/)**

### 适合 PM 的科普

5. **播客《三五环》或《乱翻书》中 LegalTech / 企业法务数字化相关单集（按平台搜索「合同 AI」「法务科技」）**

### 动手小练习（30 分钟内）

**练习：给一份 5 页《保密协议（NDA）》设计「最小审查清单」**

1. 找一份公开 NDA 模板（或公司内部模板脱敏版）；
2. 列出 10 条检查项，分成「规则可判」（如保密期限年数、管辖地）和「需人判断」（如保密信息定义是否过宽）；
3. 标出每条误报/漏报的业务后果（高/中/低）；
4. 写一句「人工必须介入」的触发条件。

**目标**：体验 PM 在合同 AI 里最有价值的交付物不是 prompt，而是**可运营的风险清单与审批策略**。
