# 2. 业务场景故事

## 《那份藏在 PDF 角落里的八年老兵》

### 第一章：周三下午四点，八百份简历只够筛一轮

周三 16:00，SaaS 公司「云栈」要在一周内补齐 15 个后端岗位。招聘负责人程薇盯着后台：周末校招专场涌进 812 份简历，用人经理老周只肯在周五前看 30 份初筛名单。

「人工全看一遍，两个人得干三天。」程薇对产品经理陆远说，「我不管你怎么做，周四晚上八点前，给我一份能解释的短名单。」

陆远刚上线的「智能简历筛选」承诺三件事：自动解析简历、按 JD（职位描述）打分排序、把明显不符的挡在门外。程薇的 KPI 很直白——**初筛耗时从 48 小时压到 6 小时，用人经理投诉率不能涨**。

当晚，系统跑完第一轮。812 份变成 67 份「推荐」、198 份「待定」、547 份「不匹配」。程薇松了口气，把 Top 30 发给老周。老周扫了一眼：「怎么一个做过支付核心系统的人都没有？你们是不是把‘有分布式经验’理解成‘写过微服务 demo’了？」

程薇心里一沉——名单看起来很「整齐」，却少了某种她凭直觉才会注意的东西。

### 第二章：机器读简历，读丢的不只是错别字

陆远拉出被拒样本。排名第一的「不匹配」理由很刺眼：候选人方敬，八年支付清算经验，简历标注「缺少 Kubernetes 生产环境经验」。

方敬的简历是设计师排版的 PDF：左侧窄栏写技能标签，右侧正文才是项目细节。解析模块（把文件变成结构化字段的程序）把左侧「技能：Java、MySQL」当成了全文，右侧「主导华东清算中心 0 到 1 重构，日峰值 8000 万笔」几乎没被抽出来。

「这不是模型笨，是输入就没喂对。」陆远指着解析日志，「版式识别失败，后面打分全建立在空壳字段上。」

团队连夜补了版式检测：双栏、表格、图片嵌入简历先走专用解析；解析置信度（程序对自己这次抽取有多把握）低于 60% 的，不直接拒，改入「待人工确认」队列。

周四清晨重跑，方敬进了待定区，分数从 23 升到 81。程薇刚想庆祝，老周又扔来一句：「待定里怎么还有三个只写过 ToB 实施、没碰过代码的？你们是不是为了凑数把门槛放水了？」

——解析修好了，排序逻辑却开始失控。

### 第三章：以为调高召回就行，结果经理更忙了

陆远面临第一个产品二选一：**初筛要「少漏好人」，还是「少打扰经理」？**

算法同学主张放宽：把「推荐」阈值从 75 分降到 62 分，宁可多推，也别漏掉方敬这种硬货。程薇点头——周五 deadline 在前，漏人比多推更致命。

重跑后，推荐池从 67 扩到 143。老周半小时就炸了：「30 个名额，你塞给我 80 个‘可能行’，等于没筛。」

更麻烦的是投诉来自内部。一位女工程师私下问程薇：「为什么和我学校、经历相近的同事都进了待定，我却直接被标‘不匹配’？」陆远调出特征贡献（模型判断时各字段占多大权重）：「统招本科」权重意外偏高，「女性」在历史录用数据里出现频率低，模型间接学到了偏见。

程薇在战情会上拍桌：「我们可以快，但不能把不合法的风险也自动化。」

陆远被迫加了两条硬规则：任何「不匹配」必须带可展示理由；涉及学历、性别、年龄、婚育等敏感字段不得进入自动否决因子。推荐名单压缩回 45，但每条附「匹配点 / 风险点」摘要，经理解释成本反而降了。

——他们以为「降阈值」能救场，真正的问题是**没有分层**：把「能不能读」和「该不该推」混成了一道闸。

### 第四章：48 小时倒计时里的降级与妥协

距 deadline 还剩 18 小时，外部语义匹配服务（根据经历含义而非死磕关键词判断岗位契合度）抖动，响应从 200 毫秒飙到 8 秒，超时率 12%。

脆弱方案是全站暂停；危险方案是超时默认「通过」。陆远选了中间路：

- 解析成功的简历，先用**规则层**做硬过滤：工作年限、必备技能、工作地点；
- 语义打分超时则标记「待算分」，进缓存队列重试，不直接定生死；
- 重试三次仍失败，进入「人工速览」——只看解析出的项目摘要，由招聘助理 90 秒内做是/否/待定；
- 所有自动「不匹配」改叫「建议不匹配」，经理和 HR 可一键翻案，翻案记录回灌训练集。

程薇带着两名助理啃「人工速览」队列，陆远盯面板：自动处理 81%，人工兜底 19%，平均单份成本 0.8 元 API 费加 1.2 分钟人力。比纯人工便宜，但比「全自动化」贵——**可解释的代价，是算力和人力的双账**。

周四 20:10，方敬出现在推荐榜第 4，附注：「支付清算核心经验与岗位高度相关；K8s 生产经验不足，建议面试重点考察容器化迁移方案。」老周回复：「这个可以聊。」

程薇以为收官了。

### 第五章：周五早上，真正翻车的是「没进来的那个人」

周五 09:15，程薇收到一封外部邮件。候选人唐溯投诉：「系统显示我不匹配，理由是没有大型项目经验。」

唐溯的「大型项目」写在附件作品集里——一份 40 页的 PDF，主简历只有链接。解析器读了主简历，没追附件；规则层把「无项目描述」判为不匹配。

更糟的是，唐溯是残障人士，作品集用语音转写生成，段落格式混乱，OCR（光学字符识别，从图片或扫描件里读字）置信度低。若按周三版系统，他会在「低置信」通道被挡；周四改版后，低置信进了人工速览，但招聘助理按「缺项目」快速点了不匹配。

陆远调出全链路：不是模型歧视，是**多文档简历没有纳入同一候选人视图**；人工速览为了赶量，没看到作品集链接。

第二个产品二选一摆在桌上：**继续追求周四晚交付的 30 人名单，还是暂停 2 小时补「附件聚合 + 速览强制展开链接」？**

程薇选了后者。她给老周打电话：「名单晚 2 小时，但我不想下周在劳动仲裁里解释‘算法拒了残疾人’。」

补丁上线后，唐溯经历被重新评估：项目经验充分，技能栈略宽，落入「待定」。程薇亲自致电道歉，邀请走人工面试通道。唐溯答应用时，但要求书面说明筛选逻辑——这封说明后来成了云栈对外公示的「AI 招聘辅助说明」初稿。

周五 14:00，老周收到 28 人推荐、9 人待定、5 人「建议不匹配需 HR 复核」。他没有再骂「凑数」，只说：「至少我能看懂你们为什么推这个人。」

程薇的周报写下一句：「AI 没替我们决定雇谁，但它逼我们把‘凭什么不雇’说清楚。」

——而陆远在复盘里加了一条：**招聘 AI 的终点不是自动发 offer，而是让每一次拒绝都经得起问。**

# 3. 系统正解

简历筛选系统的数据流是：多格式接入与版式检测 → 文档解析与字段抽取（含附件聚合）→ 硬规则过滤（年限、地点、必备项）→ 语义匹配与评分 → 风险与偏见审计 → 分层输出（推荐 / 待定 / 建议不匹配）→ 人工速览与经理复核 → 反馈回灌。第一章的 deadline 压力对应编排与 SLA；第二章方敬丢经历对应解析模块；第三章阈值与偏见对应评分策略与合规护栏；第四章超时对应降级与人工兜底；第五章唐溯对应多文档视图与速览质检。系统目标不是「替 HR 做录用决定」，而是在高压下把「可读、可排、可解释、可翻案」做到位。

# 4. 类比锚定

把这套系统类比成「医院分诊台 + 化验单解读」，而不是「自动开刀的医生」。

| 医院分诊（旧经验） | 简历筛选系统（新系统） | 映射关系 |
|---|---|---|
| 挂号台核对身份与主诉 | 简历接入、格式校验、岗位绑定 | 先确认「这份材料算数、对应哪个岗」 |
| 护士量体温、血压 | 解析出年限、技能、项目等结构化字段 | 把非结构化描述变成可计算信号 |
| 急症硬指标（过敏史、孕周） | 必备技能、工作地点等硬规则 | 一票否决项，不交给模糊打分 |
| 专科会诊意见 | 语义匹配与岗位契合评分 | 处理「经历含义相近但表述不同」 |
| 分诊颜色（红/黄/绿） | 推荐 / 待定 / 建议不匹配 | 把有限注意力导向不同处理强度 |
| 医生最终问诊 | 用人经理面试与录用决策 | 关键决定权在人，不在模型 |
| 病历留痕与复诊 | 打分理由、翻案记录、审计日志 | 事后可追溯、可改进 |

# 5. 类比边界

## 系统要点

- **解析优先于打分**：版式失败、附件遗漏、低置信抽取未处理前，不应输出强硬「不匹配」。
- **分层而非单阈值**：硬规则、语义分、风险审计、人工速览是不同层，不能用一个分数包打天下。
- **解释与翻案是功能**：每条自动结论需可读理由；HR/经理可 overturn，且记录进评估集。
- **偏见与合规护栏**：敏感属性不得作自动否决因子；定期做群体通过率差异审计。

## 类比不能覆盖的机制

- 分诊处理的是**生理指标相对标准化**的信号；简历表述高度主观，同一经历可有多种合法写法。
- 医院急症有相对明确的医学协议；岗位「合适」含文化契合、潜力判断，模型只能辅助不能替代。
- 分诊错误可快速二次评估；招聘拒信往往**单向**，候选人很少能即时纠正系统误读。
- 医疗有强制报告与执业责任链；招聘 AI 的法律责任仍在雇主与流程设计者。
- 医院流量峰值的代价是等待时间；招聘漏掉顶尖候选人的代价是**机会成本**，难以用排队时长衡量。

## 失效点

- 作品集、开源链接、内推备注等**体系外信号**占主导时，纯简历解析筛选价值骤降。
- 小样本岗位（全新职类、总共几十份简历）语义模型缺乏稳定参考，硬上自动排序易过拟合 JD 措辞。
- 组织若把「不推荐」等同于「不面试」且禁用翻案，系统再完善也会变成**自动化背锅位**。

# 6. 复盘室

**程薇说**：「周四晚上我差点为了准时交名单，跳过附件聚合。按时交付很重要，但招错人的成本和拒错人的声誉更贵。」

- 真相是：招聘场景里，**时间压力常把「可解释」挤成可选**；产品要把「晚 2 小时」和「劳动合规风险」放在同一张决策表上。
- 如果你做 PM：deadline 需求要写清「可降级的环节」与「不可降级的红线」，别让研发自行猜哪些能砍。

**陆远说**：「我们一开始把问题定义成‘怎么让模型更准’，其实先是‘怎么让简历被完整读到’。」

- 真相是：B 端文档 AI 里，**解析失败导致的错杀**，往往比模型打分错误更常见、更冤枉。
- 如果你做 PM：验收先看「字段抽取完整率 / 低置信占比 / 版式分层覆盖率」，再谈排序 AUC。

**老周说**：「我不需要 AI 替我决定雇谁，我需要它别浪费我时间，也别替我做蠢拒绝。」

- 真相是：用人经理要的是**可行动的短名单 + 可质疑的理由**；「推 80 个人」和「推 30 个说不清的人」一样糟糕。
- 如果你做 PM：对业务方承诺「自动化率」时，同步承诺「经理解释时间中位数」和「翻案率」——否则效率指标会骗人。

# 7. 学习资料

1. **《Structured Resume Parsing: Challenges and Approaches》类技术综述（适合：要和算法对齐输入质量的 PM）** — 理解双栏 PDF、表格、扫描件为何导致字段丢失，便于写解析验收标准。
2. **EEOC（美国平等就业机会委员会）关于 AI 招聘工具的合规指引（适合：负责 HR Tech 合规的产品）** — 了解自动化筛选在歧视风险上的监管关注点，即便做国内产品也宜建立审计习惯。
3. **LinkedIn / Greenhouse 等厂商公开的「候选人排序透明度说明」产品文档（适合：写对外说明与经理面功能的 PM）** — 参考业界如何展示「为何推荐、如何申诉、人保留最终决定权」。
4. **People Analytics 领域关于「简历筛选偏见审计」的实践文章，如 MIT Sloan 相关案例（适合：要设计偏见检测看板的 PM）** — 学习按院校、性别、年龄分组对比通过率，而不只盯整体准确率。
