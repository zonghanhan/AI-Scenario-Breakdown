- **难度**：进阶

- **前置知识**：建议先读「RAG 知识库问答系统」，理解「先查资料再回答」；本篇把同一套「查—想—做」延伸到**真正动手调用业务系统、串联多步操作**

昨天你刚搞懂 RAG：用户问一个问题，系统先检索知识库，再组织答案。今天进入 Tier 3 的第二块：**当用户说的不是「告诉我答案」，而是「帮我把这事办完」——系统怎么拆任务、调工具、记状态、出错回退，以及什么时候必须停下来问人。** 这就是企业里常说的 **Agent**（智能体：能根据目标自主规划步骤、调用外部工具、并根据中间结果调整下一步的系统）。

# 2. 业务场景故事

## 《那句「你帮我把续签办一下」》

### 第一章：销售总监在群里甩了一句话

周四上午 9:42，工业耗材 SaaS 公司「链材」的华东销售总监老周在企业微信群里 @ 了渠道运营专员小夏：

> 「小夏，帮我把苏州恒拓的续签办一下。去年 Q4 他们达标了，合同 3 月 31 日到期，别耽误。模板用标准经销协议，折扣跟去年一样。」

小夏刚入职四个月，脑子里立刻冒出一串待办：查恒拓是不是现有客户、Q4 销售额到底达没达标、去年合同折扣是多少、续签要不要走特批、合同模板用哪一版、审批链找谁……她打开 CRM、BI 看板、合同系统、OA 审批，五个系统来回切，光找入口就花了 12 分钟。

隔壁工位的师兄说：「你不是在试点那个『渠道运营助手』吗？直接跟它说。」

小夏把老周的原话粘进助手对话框，加了一句：「请帮我启动续签流程。」

3 秒后，助手回复：

> **已理解您的需求。我将为您处理「苏州恒拓经销合同续签」任务，预计执行以下步骤：**
> 1. 查询客户档案与当前合同状态
> 2. 核验 Q4 销售达标情况
> 3. 读取续签政策与折扣规则
> 4. 生成续签合同草稿
> 5. 创建 OA 审批单并通知相关审批人
>
> **请确认是否继续？** [确认执行] [修改条件] [取消]

小夏愣了一下——她本来只想少点几个系统，没想到助手先把「要干什么」列出来了，还等她点头。

**悬念**：这看起来比她自己跑腿省事。但「确认执行」之后，系统里到底会发生什么？

---

### 第二章：第一步查客户，就撞上了「同名不同司」

小夏点了「确认执行」。助手界面右侧展开一条实时执行日志（类似快递物流轨迹）：

```
[09:43:01] 步骤 1/5：查询客户档案 … 调用 CRM.search_customer
[09:43:02] 命中 2 条结果：苏州恒拓贸易有限公司 / 苏州恒拓机电设备有限公司
[09:43:02] 置信度不足，暂停执行，等待澄清
```

助手弹出追问：

> 我在 CRM 里找到两家名称含「苏州恒拓」的客户：
> - **A**：苏州恒拓贸易有限公司（经销商，合同编号 HT-2024-0387，到期日 2026-03-31）
> - **B**：苏州恒拓机电设备有限公司（终端客户，无在途经销合同）
>
> 根据您提到的「续签」「去年合同」，我推测是 **A**，但为避免操作错误，请确认。

小夏选了 A。她心里一紧：要是没有这个暂停，后面所有步骤都可能跑在错误客户上——续签错公司，比「没办成」麻烦十倍。

产品经理林牧（就是之前做 RAG 问答的那位）后来解释：这不是模型「不够聪明」，而是**产品故意设计的硬闸门**。涉及写操作（创建合同、发起审批、发通知）之前，身份歧义必须清零。Agent 可以猜，但**不能带着猜的结果去改生产数据**。

**悬念**：客户确认了，达标情况好查吗？

---

### 第三章：达标数据在 BI 里，规则却在知识库里

```
[09:43:18] 步骤 2/5：核验 Q4 销售达标 … 调用 BI.get_dealer_performance
[09:43:21] 苏州恒拓 Q4 回款额：¥ 4,827,000（达标线 ¥ 4,500,000）✓
[09:43:22] 步骤 3/5：读取续签政策 … 调用 KB.retrieve + PolicyEngine.match
```

到这里还算顺利。BI 返回恒拓 Q4 回款 482.7 万，超过 450 万达标线。接着助手去知识库拉《2025 年经销续签管理办法》，并用规则引擎匹配：达标客户、标准续签、折扣不高于上一年度。

问题出在折扣上。

```
[09:43:29] 步骤 3/5 警告：去年合同折扣 18%，但 2025-01-15 起华东区启用新折扣上限 15%
[09:43:29] 冲突：用户指令「折扣跟去年一样」vs 政策「新签/续签不得超过 15%」
[09:43:29] 降级：生成两个合同版本供选择，并标记需销售总监特批
```

助手没有擅自把 18% 写进合同，也没有默默改成 15% 假装没事。它做了三件事：

1. 在合同草稿里生成 **15% 标准版** 和 **18% 特批申请版** 两个附件；
2. 在 OA 审批单备注里写明冲突原因和依据文档；
3. 把任务状态改成 **「待人工决策」**，@ 老周确认是否走特批。

小夏第一次意识到：这个助手不是「聊天框里看起来很聪明」，而是一套会**调用 CRM、BI、知识库、合同系统、OA 五个工具**的执行器。RAG 只负责「告诉你政策怎么写」；Agent 还要判断**当前这条政策是否覆盖这个客户、是否与用户指令冲突、冲突时谁有权拍板**。

**悬念**：政策冲突还能兜底。如果合同已经创建了一半，下一步 OA 又超时了呢？

---

### 第四章：审批单创建成功，但通知发不出去

老周在手机上点了「18% 特批，按去年执行」。助手继续：

```
[09:51:03] 步骤 4/5：生成续签合同草稿 … 调用 ContractGen.create_draft ✓
[09:51:17] 步骤 5/5：创建 OA 审批 … 调用 OA.create_approval ✓ 审批单号 OA-2026-04182
[09:51:18] 步骤 5/5：发送通知 … 调用 IM.notify_approver ✗ 失败（接口 503）
[09:51:18] 重试 1/3 … 失败
[09:51:25] 重试 2/3 … 失败
[09:51:38] 重试 3/3 … 失败
[09:51:38] 进入异常队列，任务状态：部分完成
```

小夏看到界面上的状态不是绿色的「已完成」，而是黄色的 **「部分完成 — 审批已创建，通知未送达」**。

助手没有回滚已经创建的合同草稿和 OA 单——因为那些步骤成功了，且都有业务 ID；它做的是：

- **保留已成功步骤的产物**，在任务详情里展示 OA 单号、合同草稿链接；
- **失败步骤进入可重试队列**，不会无限自动重试（避免重复发 30 条消息给审批人）；
- **生成人工待办**：「请手动通知法务张敏审批，或点击此处重试通知」；
- **全链路留痕**：哪一步调了哪个工具、入参出参、失败原因、重试次数，全部可审计。

林牧说：「Agent 最怕两种极端设计：一是任一步失败就全部回滚，审批单明明建好了却被删掉；二是失败了就假装成功，销售以为法务收到了，其实没有。」

正确的做法是 **按步骤粒度管理状态**（这叫「 Saga 式补偿」：每一步成功就提交，失败只补偿或重试失败的那一段，而不是把前面正确的操作一笔勾销），并让人类清楚知道「办到了哪一步」。

**悬念**：小夏手动转发链接给法务后，这事就算完了吗？老周问：「这玩意儿靠谱吗，下次我能直接让它『看着办』吗？」

---

### 第五章：复盘会上，林牧画了一张「执行轨迹图」

周五复盘，林牧把这次任务做成一张时间线：

| 时间 | 动作 | 类型 | 结果 |
|---|---|---|---|
| 09:43:01 | CRM 查客户 | 只读 | 歧义 → 人工确认 |
| 09:43:18 | BI 查达标 | 只读 | 通过 |
| 09:43:22 | 知识库 + 规则引擎 | 只读 | 发现折扣冲突 |
| 09:50:41 | 销售总监确认特批 | 人工决策 | 18% |
| 09:51:03 | 创建合同草稿 | **写操作** | 成功 |
| 09:51:17 | 创建 OA 审批 | **写操作** | 成功 |
| 09:51:18 | IM 通知审批人 | **写操作** | 失败 → 人工补发 |

老周问：「所以它能自动办，但不能完全放飞？」

林牧点头：「对。这个 Agent 的决策权限是分层的：」

- **只读查询**（查客户、查数据、查政策）：可自动执行，歧义时暂停追问；
- **草稿生成**（合同预览、审批单草稿）：可自动，但需标记「未生效」；
- **生效写操作**（正式提交审批、对外发邮件、改客户主数据）：必须有确认闸门或人工审批；
- **高风险操作**（改折扣、延期付款、作废合同）：即使用户说「看着办」，也不能跳过授权链。

算法同事补充：「模型负责的是**理解意图、拆步骤、决定下一步调哪个工具**；工具本身有硬权限，模型调不了它不该调的东西。」

小夏算了笔账：自己跑一遍至少要 35–40 分钟，还容易漏查新折扣政策；助手全程 8 分钟，其中 6 分钟是在等她和管理员确认。**省下的不是打字时间，是跨系统核对和「怕办错」的心理负担。**

到这儿，故事里的「渠道运营助手」已经露出真面目：它不是更会聊天的 RAG，而是一套 **Agent 多步骤任务执行系统**——能规划、能调用工具链、能维护任务状态、能在异常时降级和补偿。

### 第六章：第二次请求，它差点帮了倒忙

试点第二周，有销售在助手里说：

> 「把苏州恒拓的合同撤了，他们不想续了。」

助手没有立刻执行，而是先问：

> 您说的「撤了」是指：
> 1. **撤回审批中的续签申请**（OA-2026-04182，当前待法务审批）
> 2. **作废已签署合同**（需法务 + 总监双签，且涉及已发生订单）
> 3. **仅标记客户「本年度不续签」**（不影响在途审批，需填写原因）
>
> 另外，检测到该客户有 ¥ 126 万应收账款未结清。作废合同可能影响回款条款。**请确认具体意图。**

销售选了「撤回审批中的续签申请」。助手执行：

```
调用 OA.withdraw_approval(OA-2026-04182) ✓
调用 ContractGen.archive_draft(draft-8831) ✓
通知发起人：审批已撤回
任务完成
```

如果是「作废已签署合同」，系统会要求上传客户书面确认、触发法务复核，**不会**因为一句口语就动核心合同数据。

林牧在文档里写下产品原则：**Agent 的「自主」体现在自动拆步骤和串联工具，不体现在替用户承担法律与商业后果。** 用户说得模糊时，宁可多问一句，不可错办一件事。

# 3. 系统正解

这套系统的核心数据流是：**意图理解 → 任务规划（拆步骤）→ 工具选择与调用 → 状态机更新 → 观测/审计 → 异常降级与人工闸门 → 结果交付**。故事第一章的「列出五步等你确认」对应规划模块；第二章 CRM 双名命中暂停对应**歧义消解与写操作前闸门**；第三章知识库 + 规则引擎对应 **RAG 与确定性规则的组合**（Agent 不等于纯大模型瞎跑）；第四章 OA 成功但通知失败对应**按步骤粒度的状态管理与补偿重试**；第五章权限分层对应**工具级鉴权与风险分级**；第六章「撤了」追问对应**意图消歧与高风险操作保护**。

- **写需求文档时**：不要只写「支持自然语言办事」，要列出可用工具清单、每步输入输出、哪些步骤只读/可写/需人工确认、失败后的状态与补偿策略。
- **做方案评审时**：问清楚「规划错了怎么办」「工具调用超时是否重试」「部分成功如何展示」「用户能否看到执行轨迹与回滚入口」。
- **设计指标时**：看任务完成率、**一步成功率**、人工介入率、平均步骤数、单任务 Token/工具调用成本、以及「错误执行率」（比「没办成」更危险的指标）。
- **跨团队沟通时**：与算法确认规划边界与工具描述格式，与工程确认幂等、超时、并发和审计日志，与业务方确认哪些操作绝对不能自动执行。

# 4. 类比锚定

可以把 Agent 多步骤任务执行类比成**「资深行政助理帮老板跑一趟跨部门流程」**。老板一句话「把续签办了」，助理不会立刻冲进法务室盖章，而是先搞清是哪个客户、翻政策、填表、找审批、送文件、对方不在就留言回头再补——全程记着办到哪一步了。

| 行政助理办事（旧经验） | Agent 多步骤任务执行（新系统） | 对应关系为什么成立 |
|---|---|---|
| 听懂老板口头交代 | 意图理解与槽位填充 | 都要把模糊自然语言变成结构化任务参数 |
| 列出「我先去 A 处查，再去 B 处盖章」 | 任务规划 / Plan | 都是把目标拆成有序子步骤 |
| 去档案室、财务、法务各部门窗口 | 工具调用（CRM、BI、OA、IM） | 每一步依赖外部系统能力，不是自己凭空完成 |
| 记事本记录「表已填，还差张总签字」 | 任务状态机 / 执行上下文 | 多步任务必须持久化中间状态，支持暂停与恢复 |
| 发现两个「恒拓」，打电话跟老板确认 | 歧义消解 / 人工闸门 | 关键信息不确定时，不能替决策者拍板 |
| 文件已递交但张总没收到微信 | 部分成功 + 补偿重试 | 已完成步骤保留，只对失败环节补救 |
| 涉及解约/付款变更，必须请老板当面确认 | 高风险写操作授权 | 越权风险越高，自动化程度越低 |

# 5. 类比边界与系统要点

## A. 系统核心要点总结

- **数据输入要求与格式规范**：除自然语言外，还需会话上下文（渠道、角色、客户 ID）、工具 Schema（每个工具叫什么、要什么参数、返回什么）、以及权限令牌；工具描述写不清，Agent 就会频繁选错工具或传错参。
- **AI 模型的能力边界与置信度**：大模型擅长理解与规划，不保证每步规划最优；应用**规划—执行分离**，关键判断用规则引擎或业务服务兜底，并对低置信规划触发人工确认。
- **人工介入的触发条件与流程**：身份歧义、政策冲突、超额授权、高风险写操作、连续工具失败、用户意图多义——都应触发「暂停—展示证据—等待决策」，而非默认继续执行。
- **系统的容错与降级机制**：工具超时要有重试上限与退避策略；部分成功要明确展示；必要时提供一键重试、转人工、以及针对写操作的补偿（撤回、作废草稿、冲正）。
- **成本结构**：成本 = 每次规划的模型调用 + 每步工具 API + 状态存储 + 人工兜底工时；步骤越多、规划轮次越多，Token 消耗越高；「一次对话办完」的单价可能远高于单次问答。
- **可观测与审计**：必须保留完整执行轨迹（计划、工具名、入参、出参、耗时、决策点），否则出问题时无法定责，也无法优化工具描述和规划策略。

## B. 类比在哪里会失效

1. **真人助理有常识且知进退，Agent 的「常识」来自训练与提示，边界不可预测。** 同一句「撤了」在不同上下文可能指不同操作；助理会凭经验追问，Agent 若闸门设计不好可能猜错并执行。
2. **行政流程大多线性，Agent 执行可能是动态重规划。** 查到折扣冲突后，后续步骤从「直接生成合同」变成「先走特批再生成」；步骤数与顺序在执行中会变，不能用固定 checklist 类比。
3. **助理办错了要担责，Agent 办错了责任在系统设计方。** 产品必须假设「一定会错」，用权限、确认、审计来限制爆炸半径；不能因「像人一样聪明」就放松风控。
4. **助理一天能跑有限件事，Agent 并发高但共享下游系统配额。** 批量「帮我都办了」可能打爆 OA、合同系统接口；需要队列、限流、优先级，类比里「窗口排队」的规模效应被低估。
5. **助理会学习老板习惯，Agent 的「学习」需要显式反馈与版本管理。** 改工具、改政策、改模型后，旧任务轨迹可能不再适用；类比无法覆盖持续迭代、A/B 与回滚问题。

# 6. 深度业务洞察

1. **业务侧：用户要的是「少跑腿」，不是「少点确认按钮」。** 若为了炫技把确认全去掉，一次误续签的成本远高于多点三次「确认」。立项时应与业务方逐条划定「可自动 / 需确认 / 禁止自动」清单，而不是追求「全自动」口号。

2. **技术侧：最难的不是让模型会调 API，而是工具设计与状态一致性。** 工具粒度过粗（一个 `handle_renewal` 黑盒）出错了无法定位；过细（五十个微工具）规划易迷路。CRM 客户 ID 与合同系统客户编码不一致、OA 单已建但助手以为失败——这些**跨系统 ID 映射与幂等**才是工程深水区。

3. **运营侧：冷启动期要有人类「陪跑」标注。** 哪些问法应触发歧义、哪类冲突应走特批、哪步失败该转人工——需要从高价值任务开始，积累执行轨迹，迭代工具描述和闸门规则；没有轨迹数据，Agent 无法像 RAG 那样先离线评测再上线。

4. **合规侧：写操作必须可追溯、可撤回、可定责。** 监管与内控会问：谁授权 Agent 创建了这笔审批？依据哪条政策？模型版本与提示词是什么？若不能回答，财务与法务会禁止接入生产系统。

5. **商业侧：ROI 别按「替代一个全职助理」算，要按任务类型拆开算。** 查数据类步骤节省明显；涉及特批、谈判、例外处理的步骤仍依赖人。定价应按任务复杂度、工具调用次数、是否含写操作分级，并向客户说明「部分完成」是正常状态而非故障。

# 7. 学习资料参考

## 官方文档 / API 参考

1. [OpenAI：Function calling 指南](https://platform.openai.com/docs/guides/function-calling)
   适合第一次写「工具调用」需求的 PM。重点看如何用 JSON Schema 描述工具、模型如何选择工具、以及为何要把工具权限放在服务端而不是交给模型——用它把「Agent 能干什么」改写成可验收的接口列表。

2. [Anthropic：Tool use（Claude）](https://docs.anthropic.com/en/docs/build-with-claude/tool-use)
   适合评估多模型方案的 PM。看中英对照的工具定义、并行调用、以及「模型建议但系统执行」的分工，理解产品层必须保留的执行闸门。

3. [LangGraph 概念文档（中文社区译介）](https://langchain-ai.github.io/langgraph/)
   适合参加 Agent 技术评审的 PM。重点看「状态图」「检查点」「中断与恢复」——对应故事里的部分成功、人工介入与任务暂停，不必读代码，建立与工程对话的共同语言即可。

## 行业最佳实践

4. [微软 Azure：AI Agent 设计模式](https://learn.microsoft.com/zh-cn/azure/architecture/ai-ml/guide/ai-agent-design-patterns)
   适合画架构图、写方案背景的 PM。涵盖规划器、工具调用、人机协同、多 Agent 协作等模式，帮助判断你的场景是否需要「一个大脑」还是「多角色分工」。
